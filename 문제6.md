### 1-2. 문제 B 서술 정밀화 (결론 불변)

## 2. 합의 단서 3건 (문제5에 추가되는 구속 조항)

### 단서 ① 라이브 크론 배포 절차 (1단계 리팩토링 필수 규칙)

이 저장소는 크론이 10분마다 워킹트리를 직접 실행한다. `rank_policy.py` 분리 중 회차가 걸리면 게시가 깨진다.

- 다중 파일 리팩토링(1단계·4단계)은 **크론 휴지 시간(주말 또는 평일 20:52~익일 09:00)에만** 수행한다. 1단계는 **주말(7/12~13 일요일까지) 완료를 권장** — 월요일 09:01 첫 회차와 forward 시작일(7/13)이 새 구조로 정렬되도록.
- 부득이 장중 수정 시: 신규 모듈 파일을 먼저 완성 → py_compile → radar.py의 import 전환을 한 번의 편집으로 → 즉시 py_compile + 단독 실행 스모크 → 다음 회차 로그 확인. 편집 시작 전 `date`로 다음 회차까지 남은 시간 확인(5분 미만이면 대기).
- 각 커밋 직후 publish 로그 1회차 정상 확인을 커밋 완료 조건에 포함한다.

### 단서 ② 적대 코드리뷰 2회 대상 확대

문제5는 4단계(스캔 감사)만 2회 리뷰로 지정했다. **2단계·3단계는 실계좌 자동매매 파일(`autotrade_common.py`·`autotrade_executor.py`)을 접촉하므로 기록 전용이라도 2회로 격상한다.** 리뷰 관점 고정: "추가된 기록 코드가 어떤 예외 경로에서도 주문 결정·수량·안전필터에 영향을 줄 수 없는가"(기록 실패는 전부 fail-safe로 주문 진행).

### 단서 ③ scan_history 용량 정책

- 일별 compact 파일 **1MB 상한**(추적 후보 수백 종목 × 요약 필드 기준 여유). 초과 시 minute_summary 등 부피 필드부터 축약하고 초과 사실을 파일 메타에 기록.
- 90일 경과 파일은 분기별 아카이브(tar.gz)로 롤오버할 수 있게 경로 규약만 지금 정의(`data/radar_scan_history/archive/`). 자동 삭제는 하지 않는다(소표본 보존 원칙).
- publish 커밋 시간 증가를 관찰해 회차 지연이 생기면 scan_history 커밋을 일마감 1회로 분리한다.

## 3. 확정 실행 계획 (문제5 §4의 6단계 + 단서 반영)

| 단계 | 내용 (문제5 기준) | 시행 시점 | 리뷰 |
|---|---|---|---|
| 1 | rank_policy.py 분리 + rank4-v1/effective 7/13 + precut/published rank + 원자 쓰기 + publish 순서 보존 | **주말(권장 7/12~13)** — 단서 ① | 1회 |
| 2 | rank_path + KRX/NXT 의사결정 시점 + 자동매매 원장 기록 | 주말~월요일 저녁 | **2회** (단서 ②) |
| 3 | retro/forward/킬스위치 분리 + 모집단(krx/nxt/eod) 분리 | 월요일 저녁 이후 | **2회** (단서 ②) |
| 4 | 전체 후보·탈락 사유 스캔 감사 (`radar_scan_history`) | 크론 휴지 시간 | 2회 (문제5 원안) |
| 5 | prior/retro/forward 카드·패널 표시 | 순차 | 1회 |
| 6 | rank_eval (모델 버전별 top1/top3·spearman·NDCG) | 순차 | 1회 |

- 커밋은 문제5 §8의 8분할 그대로. 각 커밋 = 독립 롤백 가능 + py_compile·스모크·dry-run·`radar.json` 순서 diff 무변·웹 build + **publish 라이브 1회차 확인**(단서 ①).
- 정렬 순서 diff가 1건이라도 생기면 즉시 중단·보고 (문제5 §8 배포 원칙 동일).
- forward 시작일(7/13) 전에 1단계가 완료되지 않으면 effective date를 실제 배포 완료 다음 거래일로 미루고 문서·코드 양쪽을 같이 수정한다 — **"코드보다 이른 effective date" 금지.**

## 4. 최종 문장

문제5의 진단(A~D)과 설계를 검증 후 전부 수용한다. 서산 7/10 표본의 정확한 실체는 "정렬4 미운영"이 아니라 **슬롯별 모델이 갈린 오염 표본**이며, 그래서 forward 시작일 7/13 고정이 옳다.

> **정렬은 바꾸지 않는다. 대신 이번 주말부터: 순위 정책에 버전을 새기고(7/13 rank4-v1 발효), 의사결정 시점(KRX 15:18/NXT 19:50)의 순위를 영구 보존하며, 탈락 후보와 그 이유를 기록하고, prior·소급·전진을 분리해 — 7월말에는 "이 정렬이 실제로 익일 최고 고가를 맞혔는가"를 모델 버전별 숫자로 회장님께 보고할 수 있게 만든다.**

승인 시 단서 ①에 따라 1단계를 주말에 착수한다.
