### 검증 중 추가 발견 (문제3에 없던 내 보강 2건)

- **F의 B3 역전에는 기계적 설명이 있다**: 저점매집은 밴드(−10%↓) 게이트 신호라, **마감 전 밴드 이탈(=주가 회복) 종목이 dropout으로 분류**된다. 즉 B3 dropout의 호성적(100%/+17.8)은 "회복 중인 종목"이라는 구조적 편향이지 노이즈가 아니다 — 모집단 분리(P5)의 필요를 강화하는 동시에, **dropout 통계를 실전 기대치로 오독하면 안 되는 이유**(마감 시점엔 카드에 없어 살 수 없었던 종목)이기도 하다. 카드 expected는 반드시 final 모집단.
- **H의 winner rank 분포(1,1,1,3,3,3,6,7)는 새 정렬의 비교 기준선**으로 박제할 가치가 있다 — 7월말 rank_eval 도입 시 "구정렬 top3 6/8"이 베이스라인.

### 뉘앙스 정정 1건 (사실 오류는 아님)

문제3 §2.3의 도식("유의하지 않다 → 아무 조치도 하지 않는다")은 문제2의 입장이 아니다 — 문제2는 B10·B3을 강등 감시 대상으로 등재하고 킬스위치를 유지했다. 다만 문제3의 재정식화("소표본 역전은 자동 재정렬의 근거가 아니라 **상신 누적 증거**")가 회장님 전제를 더 정확히 표현하므로 **그 문구를 최종 채택**한다. "통계적으로 유의하지 않다"는 표현은 이후 문서에서 "자동 재정렬 요건 미충족(누적 증거로 계속 기록)"으로 대체한다.

## 3. 최종 실행안 (확정 — 승인 시 이 순서로 구현)

### P0. 정렬 뼈대 동결 (코드 무변경)
rank_bucket 순서·버킷 정의·자동매매 정책 불변. 소표본을 이유로 버킷 삭제·숨김 금지.

### P1. 진짜 forward 기록 + 저비용 결함 동시 수정 (1차 작업 묶음)
- `rank_model_version`(예: `rank4-v1@7f3acab`) suspect·history 영구 저장. 신호 당시 `rank_at_signal`(게시 순위)도 저장.
- `rank_bucket_stats` → **`_retro`(소급 재분류)/`_forward`(저장값 전용)** 분리. **forward 킬스위치는 저장된 신호 시점 버킷만 읽는다**(킬스위치 계약 회복 — 최우선 사유).
- 같은 커밋에 저비용 5건 동시 적용: ① history 원자적 쓰기(임시파일+fsync+`os.replace`) ② publish 재조립을 원 배열 순서 유지로 + 전후 순서 회귀 테스트 ③ expected_high 평균/중앙 분리 저장(`avg_`/`median_`/`min_next_high_pct`)·카드 표기 정정 ④ 스냅샷에 윌슨 하한·모집단·기준일 포함 ⑤ 패널에 n·고유종목·"forward 수집 중" 병기(숨김·자동제외 금지).

### P2. 전체 후보·탈락 사유 스냅샷 (축소판)
일마감 1회 `data/radar_scan_history/YYYYMMDD.json`: 전체 추적 후보 + 게이트별 통과/탈락 + 탈락 단계·복수 사유 + 컷 전 순위 + 게시 여부 + **분봉 스파크 요약(소급 불가 — 시급)** + 상태 구분(`PASS/REJECT_RULE/MISSING_DATA/API_ERROR/OUT_OF_UNIVERSE/PUBLISH_CUT`). 데이타솔루션 유형 복원이 수락 기준.

### P3. prior/retro/forward 3분리 표시
문제3 §B의 JSON 구조 채택(`rank_prior`/`retro_stats`/`forward_stats`). 카드: "정렬근거(prior 요약) · 과거 소급 n=…, 터치 … · 정렬4 이후 n=…, 수집 중". 정렬은 prior 유지, 숫자는 출처 명시.

### P4. 신호일 상대순위 평가 (`rank_eval`)
top1_hit·top3_contains_winner·rank별 평균고가·spearman·winner의 게시 rank·후보수별 분리(1종목 날 별도) — **모델 버전별 집계**. 구정렬 기준선(top1 3/8·top3 6/8) 명기.

### P5. final/intraday/dropout 모집단 분리
카드 expected = `exclusive_final`. dropout 통계는 필터 검증 전용 라벨("마감 시점 매수 불가 모집단") 필수 — §1 보강 1의 오독 방지.

### P6. 회차별 rank_path
순위·핵심 피처 변화 시에만 이벤트 append + `first_top3_at` 등 요약 필드.

### 데이터 계약 (문제3 §6 전면 채택)
prior와 실측 혼합 금지 · n=1도 저장·표시 · 조회 실패와 조건 탈락 구분 · 신호 당시 값 불변(`live_*`/`*_at_signal`) · 정렬 변경은 후보안 산출→회장님 승인→모델 버전 증가.

### 완료 검증 기준
문제3 §7의 14개 항목 전부 + 1개 추가: **⑮ 스모크 테스트에 forward/retro 분리 회귀 케이스(저장된 버킷≠재분류 버킷인 레코드가 forward 표에 저장값으로 집계되는지) 포함.** 각 단계 py_compile·스모크·dry-run·웹 build·적대 리뷰(P1~P3 각 1회, P2 스냅샷은 2회) 후 커밋 전 보고.

## 4. 최종 문장

문제3은 사실 오류가 없고, 문제2와의 이견 3곳(3분리·데이터 보존 순위·소표본 문구) 모두 문제3 쪽이 옳다 — 그대로 수용한다. 이로써 3자 합의가 완성됐다:

> **정렬 뼈대는 회장님 prior로 동결한다. 소표본은 하나도 버리지 않되 prior·소급·전진을 분리해 거짓 숫자를 만들지 않는다. forward 킬스위치의 계약을 코드로 회복하고(P1), 탈락 후보를 오늘부터 보존하며(P2), 정렬 조정은 모델 버전별 전진 성과를 근거로 회장님 승인 아래에서만 한다.**

승인 시 P1부터 착수한다.
