이 프로젝트의 최우선 KPI는 final suspects의 다음 거래일 장중 고가 기준 다음 두 값이다.

종가수익, 현재수익, +7/-5 최초도달은 별도 지표다. 이번 패치에서도 섞지 않는다.

## 3. 묶음 A — 재료 신선도 신호시각 보정

### 3.1 확정 원인

폭발일에 계산된 `material`은 `.explosion_registry.json`에 `cause_done=True`와 함께 보존된다. 며칠 뒤
reaccum 후보가 되면 기사와 재료를 다시 칠하지 않고 이 캡처본을 재사용한다.

이 설계는 폭발 당시 재료를 사후 뉴스로 오염시키지 않으므로 유지해야 한다. 문제는 시간이 지나면
변해야 하는 `freshness_days`까지 폭발일 값으로 동결된다는 점이다.

2026-07-20 history에서 확인된 대표 불일치:

### 3.3 순수 함수 계약

```python
def parse_evidence_datetime(value):
    """지원 형식을 KST aware datetime으로 반환하고 실패하면 None."""

def material_freshness_view(material, *, as_of):
    """입력 material을 변경하지 않고 신호시각 기준 material 복사본을 반환."""
```

`as_of`는 외부에서 주입한다. publish에서는 `out["generated_at"]`을 KST aware datetime으로 파싱해
사용한다. 현재 벽시계를 함수 내부에서 다시 읽지 않는다.

지원할 evidence 날짜 형식은 기존 `_age_days()`와 최소한 동일해야 한다.

```text
YYYYMMDDHHMM
YYYYMMDDHHMMSS
YYYY-MM-DD HH:MM:SS KST
YYYY-MM-DD HH:MM:SS
YYYY.MM.DD HH:MM
YYYY.MM.DD. HH:MM
timezone 포함 ISO-8601/RFC 형식
```

### 3.4 파생 필드

파싱 가능한 evidence 중 `as_of` 이하의 가장 최신 발행시각을 기준으로 계산한다.

```text
freshness_as_of
freshness_days
freshness
freshness_basis = evidence_latest_datetime
evidence_latest_datetime
freshness_parse_status = complete | partial | missing | future_only
future_evidence_n
captured_freshness_days
```

규칙:

1. `freshness_days = (as_of - latest_evidence_datetime).total_seconds() / 86400`이다.
2. 0 미만으로 clamp해 미래 evidence를 당일 재료로 만들지 않는다.
3. 미래 evidence는 계산에서 제외하고 `future_evidence_n`에 센다.
4. 과거 evidence 일부만 파싱되면 `partial`이다.
5. 전부 없거나 파싱 실패면 `missing`, 미래 evidence만 있으면 `future_only`이며 경과일은 null이다.
6. 기존 freshness 라벨 경계(당일~1일/2~3일/1주내/오래됨)를 그대로 사용한다.
7. `grade`, `score`, `directness`, `risk_flags`, evidence 원문은 변경하지 않는다.

### 3.5 적용 대상과 불변성

다음 두 배열의 material 보유 행 모두에 적용한다.

```text
out["suspects"]
out["blocked_suspects"]
```

blocked 카드는 추천 대상은 아니지만 화면에 재료를 표시하므로 동일한 날짜 정합성이 필요하다.

함수는 반드시 idempotent해야 한다.

- 첫 호출에서만 기존 `freshness_days`를 `captured_freshness_days`로 보존한다.
- 이미 `captured_freshness_days`가 있으면 재호출 시 덮어쓰지 않는다.
- 입력 객체와 레지스트리 파일은 변경하지 않는다.
- 같은 material과 같은 as_of 입력은 같은 결과를 만든다.

### 3.6 테스트

1. 파세코 7/13 09:26 evidence를 7/20 15:30 기준으로 계산하면 약 7.25일이다.
2. 동일 material을 서로 다른 as_of로 계산하면 evidence는 같고 경과일만 증가한다.
3. 입력 material과 레지스트리 원본 hash가 호출 전후 동일하다.
4. complete/partial/missing/future_only가 각각 구분된다.
5. 두 번 호출해도 `captured_freshness_days`가 변하지 않는다.
6. suspects와 blocked_suspects가 같은 규칙으로 보정된다.
7. 웹 게시본과 history의 보정값이 동일하다.
8. rank_bucket, published_rank, suspects 순서가 패치 전후 동일하다.

### 4.3 대상종목 검증 공용 헬퍼

투자경고와 투자위험 경로가 같은 본문 검증 함수를 사용해야 한다.

```python
def parse_notice_target(text):
    """대상종목명과 증권종류를 파싱. 불명확하면 parse_status=unavailable."""

def notice_target_matches(expected_name, target_name):
    """법인표기·공백만 정규화하고 우선주 구분자는 보존해 정확 일치."""
```

정규화에서 제거 가능한 것:

- `(주)`, `주식회사`

- `우`, `우B`

- 종목을 구분하는 숫자·영문

### 4.4 호출 시그니처와 캐시

기존 호출을 깨지 않는 optional 인자를 추가한다.

```python
fetch_release_rule(code, max_pages=8, expected_name=None)
evaluate_release_for(code, daily, current_price, *, as_of_date=None, expected_name=None)
forecast_release_for(code, daily, current_price, *, as_of_date=None, expected_name=None)
risk_release_date(code, max_pages=2, expected_name=None)
recent_risk_release(code, today_yyyymmdd, window_days=3, expected_name=None)
```

`radar.py`는 두 경로 모두 `s["name"]`을 끝까지 전달한다.

```text
evaluate_release_for(..., expected_name=s["name"])
recent_risk_release(..., expected_name=s["name"])
```

`_NOTICE_CACHE`, `_RULE_CACHE`, `_RISK_CACHE`는 모두 다음 키를 사용한다.

```text
(normalized_code, normalized_expected_name)
```

expected_name이 없으면 기존 호출은 동작할 수 있지만 대상 일치가 검증되지 않았음을 결과에 명시하고,
신규 운영 호출에서는 반드시 이름을 전달한다.

### 4.5 후보 선택 규칙

1. 공시 목록을 최신순으로 읽는다.
2. 제목상 투자경고 또는 투자위험 후보를 찾는다.
3. 이벤트를 지정/해제로 확정하기 전에 본문을 읽는다.
4. 본문 대상종목이 요청 종목과 정확히 일치하면 해당 이벤트를 채택한다.
5. 명백한 다른 종목이면 `mismatch` 증거를 남기고 다음 후보를 본다.
6. 최신 관련 후보의 대상종목을 파싱하지 못하면 `target_unparseable`로 중단한다.
7. 파싱 실패를 이유로 더 오래된 지정공시를 현재 상태라고 낙관적으로 선택하지 않는다.

`risk_release_date()`는 현재의 제목 전용 `_ROW_RE` 경로를 그대로 사용하면 안 된다. href와 notice id를
보존하는 `_notice_rows()` 계열로 통일해 본문 검증이 가능해야 한다.

하위호환 Boolean·날짜 API와 증거 보존을 함께 만족시키기 위해 내부에는 구조화된
`risk_release_event()`를 두는 방식을 권장한다. `risk_release_date()`와 `recent_risk_release()`는 이
결과의 얇은 wrapper로 유지하고, 신규 운영 행의 radar history에는 `alert_risk_release_evidence`를 함께 저장한다.
이 증거 필드도 ranking에는 사용하지 않는다.

```python
def actual_next_market_session(signal_date, benchmark_bars):
    """005930의 signal_date 이후 첫 유효 KRX 거래일. 없으면 None."""
```

원칙:

- 기준종목 조회는 평가 실행당 캐시해 종목마다 다시 호출하지 않는다.
- 기준종목의 신호일 또는 다음 거래일을 조회창에서 확인하지 못하면 평가를 보류한다.
- `resolve_next_trade_date()`의 정적 달력은 사전 표시용으로 유지할 수 있지만, 사후 평가 SSOT는 실제
  KRX 기준종목 거래일이다.
- 정적 달력과 기준종목 실거래일이 다르면 임의로 정적 달력 날짜를 강제하지 않는다. 진단을 남기고
  사후 평가는 기준종목 실거래일을 사용한다. 기준종목 데이터 자체가 불완전하면 pending이다.
- 기준종목 자체가 개별 정지일 가능성에 대비해 장기적으로는 시장 캘린더 소스를 둘 수 있으나, 이번
  최소패치에서는 거래량 양수와 날짜 연속성을 검증한다.

### 5.3 종목의 다음 거래일 상태 판정

신호일 봉, 실제 다음 시장 거래일, 그 이후 종목 봉을 함께 본다.

```text
EVALUATED
EXCLUDED_UNTRADABLE
PENDING_MARKET_SESSION
PENDING_DATA_QUALITY
EXPIRED_UNRESOLVED
```

#### A. 정지 플레이스홀더

실제 다음 시장 거래일의 종목 일봉이 다음을 모두 만족할 때만 거래불능으로 제외한다.

```text
open == high == low == close
volume == 0
```

- OHLC가 같아도 volume>0이면 정상 거래 표본이다.
- volume이 None이거나 숫자로 검증되지 않으면 `PENDING_DATA_QUALITY`다.
- volume=0인데 OHLC가 서로 다르면 데이터 불일치이므로 `PENDING_DATA_QUALITY`다.

#### B. 정지기간 뒤 복귀봉 오평가 차단

실제 다음 시장 거래일의 종목 봉은 없고, 같은 종목의 그보다 늦은 유효 거래봉이 있으면
`EXCLUDED_UNTRADABLE`이다. 늦은 복귀봉을 익일봉으로 평가하지 않는다.

#### C. 단순 봉 부재

시장 기준종목에도 다음 거래일이 아직 없으면 연휴·장전·API 조회시점 문제일 수 있으므로 기존처럼
재시도한다. 종목 봉 하나가 없다는 사실만으로 즉시 거래정지라고 단정하지 않는다.

### 5.4 history 상태 전이와 감사 증거

정상 평가:

```text
evaluated = true
evaluation_status = EVALUATED
result = 기존 익일 결과
```

거래불능 제외:

```text
evaluated = true
evaluation_status = EXCLUDED_UNTRADABLE
result = null
evaluation_exclusion = {
  reason_code,
  signal_date,
  expected_market_date,
  observed_stock_date,
  source_volume,
  source_ohlc,
  previous_result,
  corrected_at,
  logic_version
}
```

기존 잘못된 `result`는 삭제해 흔적을 없애지 말고 `evaluation_exclusion.previous_result`에 보존한다.
`collect_samples()`는 다음 하위호환 규칙을 사용한다.

```text
result 존재 AND evaluation_status가 없거나 EVALUATED -> 기존·신규 정상 표본 포함
evaluation_status가 EXCLUDED_UNTRADABLE 등 명시적 제외 -> result 유무와 무관하게 제외
result 없음 -> 성과 표본 제외
```

즉, `evaluation_status` 필드가 없던 기존 정상 평가행을 전부 탈락시키면 안 된다.

### 5.5 local next_day.json과 성과파일

history에서 제외된 행은 `collect_samples()`에서 빠지므로, 현재 samples만으로
`write_evaluation_json()`을 만들면 제외 행이 파일에서 사라진다. 다음 중 작은 구현을 선택한다.

1. history의 `evaluation_exclusion`을 별도로 수집해 next_day writer에 합치거나,
2. writer 전용으로 evaluated와 excluded를 함께 반환하는 함수를 둔다.

성과파일의 제외 건수도 정상 `samples`에서 역산하지 않는다. history의 명시적
`evaluation_exclusion`을 별도로 집계해 `write_performance()`에 전달한다. 제외 행을 분모에서 뺀 뒤
다시 통계용 samples에 섞어 카운터를 만드는 방식은 금지한다.

제외 행은 local JSON에 다음처럼 남긴다.

```text
status = excluded_untradable
next_day = null
exclusion = history의 감사 증거
populations = 신호 당시 저장값
```

성과 JSON에는 최소한 다음을 추가한다.

```text
excluded_untradable_n
excluded_halt_placeholder_n
excluded_late_resume_n
legacy_eligibility_unknown_n
```

제외 행은 +7/+13 분자와 분모, 평균·중앙 고가율, rank 성과 계산에서 모두 빠져야 한다.

### 5.6 legacy 처리

eligibility 필드가 없는 범위는 다음이다.

```text
20260714 이전 전 행
20260714 이후에도 next_session_eligibility 필드가 없는 일부 행
```

기존 파일을 이 라벨만 추가하려고 전면 재작성하지 않는다. 집계 시
`eligibility_basis=legacy_unknown`으로 계산한다. 거래불능이 원시 일봉으로 확정된 행만 위 규칙으로
제외한다.

### 5.7 모나미 005360 소급 정정

딱 다음 두 원본의 2026-07-16 신호 행만 데이터 오류 정정 대상으로 삼는다.


### 5.8 테스트

1. OHLC 동일+volume 0인 실제 다음 시장일 봉은 excluded_untradable이다.
2. OHLC 동일+volume 양수는 정상 evaluated다.

## 7. 묶음 E — +7/+13 공용 원시 판정

### 7.1 목적

### 7.2 공용 순수 함수

권장 신규 파일:

- `scripts/next_high_metrics.py`
- `scripts/next_high_metrics_test.py`

```python
def derive_next_high_metrics(entry, next_high):
    """원시 가격으로 high_pct, touch7/11/13/15, distance를 결정론 산출."""
```

가격은 `Decimal(str(value))` 또는 동등한 정확 비교를 사용한다. KRX 정수가격 기준 터치는 다음 비교와

```text
touch7  <=> next_high * 100 >= entry * 107
touch11 <=> next_high * 100 >= entry * 111
touch13 <=> next_high * 100 >= entry * 113
touch15 <=> next_high * 100 >= entry * 115
```

반환 필드:

```text
next_high_pct_raw
next_high_pct          기존 호환용 표시 반올림
touch7
touch11
touch13
touch15
distance_to_7_pct
distance_to_13_pct
metrics_version
```

touch를 먼저 원시 가격으로 판정하고 표시값을 나중에 반올림한다.

### 7.3 적용 파일

- `scripts/radar_backtest.py`
- `scripts/radar_observed_forward.py`

- 필요 시 관련 performance 타입·테스트

기존 평가행은 `entry`와 `next_high`가 있으면 네트워크 재호출 없이 파생필드만 self-heal한다. 원시 가격이
없으면 반올림된 pct로 touch를 확정하지 말고 `metrics_status=raw_price_missing`으로 둔다.
