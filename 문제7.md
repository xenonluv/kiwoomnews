# 문제7 - Mac 로컬 JSON 전량 저장 최종 확정안

작성일: 2026-07-10  
대상 저장소: `/Users/jinjin/kiwoomnews`  
문서 지위: 문제1~문제6의 합의와 Mac 로컬 JSON 전량 저장 결정을 반영한 최종 구현 SSOT

## 0. 최종 결정

레이더 연구용 원본 데이터는 **JSON만 사용해 Mac 로컬에 전량 저장**한다.

- SQL 사용 안 함
- SQLite 사용 안 함
- 별도 데이터베이스 프로그램 사용 안 함
- 권위 원본: 회차별 불변 JSON 파일
- 분봉 원본: 종목·거래일별 JSON 파일
- 의사결정 원본: KRX/NXT 시점별 JSON 파일
- 일별 색인: `manifest.json`
- Git: 웹·성과용 compact 파생 JSON만 추적
- 로컬 JSON 용량 제한 없음
- 자동 축약·자동 삭제·자동 아카이브 없음
- 백업은 회장님이 수동으로 수행

이 구조의 목적은 특정 프로그램이나 단일 파일 장애가 전체 원본을 손상시키지 않게 하는 것이다. 한 회차 JSON이 손상돼도 다른 날짜·회차·종목 파일은 그대로 보존된다.

## 1. 변경하지 않는 원칙

1. 회장님의 40년 경험칙을 현재 정렬의 강한 prior로 유지한다.
2. 현재 `rank_bucket` 순서와 조건을 변경하지 않는다.
3. 자동매매 종목 선택·rank 정책·비중·수량·안전필터를 변경하지 않는다.
4. 소표본은 n=1부터 모두 저장하고 표시한다.
5. prior, retro, forward를 혼합하거나 서로 덮어쓰지 않는다.
6. 킬스위치는 자동 판정·수동 상신이며 순위 변경은 회장님 승인 후 수행한다.
7. KRX 가격, UN 거래량·거래대금, NXT 등락률의 기준을 명시적으로 저장한다.
8. API 오류·결측과 알고리즘 조건 탈락을 구분한다.
9. 로컬 원본은 완성 후 수정하지 않는 불변 파일을 원칙으로 한다.

## 3. JSON 저장 안전 규칙

## 3.1 원자적 저장

모든 JSON 파일은 다음 순서로 저장한다.

1. 같은 디렉터리에 임시파일 작성
2. `flush()`
3. `os.fsync()`
4. 임시파일을 다시 읽어 JSON 파싱 검증
5. `os.replace()`로 최종 파일명 교체

예시:

```text
scan_090312_a1b2c3.json.tmp
-> 쓰기·fsync·파싱 검증
-> scan_090312_a1b2c3.json
```

완성된 회차 파일은 다시 열어 덮어쓰지 않는다.

## 3.2 파일명 충돌 방지

파일명에는 다음을 포함한다.

- 시각 `HHMMSS`
- 회차 고유 ID 또는 짧은 nonce

같은 초에 재실행돼도 기존 파일을 덮어쓰지 않는다.

## 3.3 스키마 버전

모든 원본 JSON 최상단에 다음을 저장한다.

```json
{
  "schema_version": 1,
  "record_type": "scan_run",
  "created_at": "2026-07-13 09:03:12 KST"
}
```

향후 필드가 추가되면 기존 파일을 수정하지 않고 새 파일의 `schema_version`만 올린다.

## 3.4 무결성 확인

각 파일에 payload checksum을 저장한다.

- 알고리즘: SHA-256
- `manifest.json`에 파일 경로·크기·checksum 기록
- secret이 포함된 원본 응답 전체를 checksum 대상으로 저장하지 않음

checksum은 손상을 발견하기 위한 것이며 암호화 목적이 아니다.

## 3.5 JSONL 사용 범위

JSONL은 append 성격이 강한 다음 데이터에만 사용한다.

- 자동매매 decision 이벤트
- 자동매매 체결 이벤트
- 선택적 운영 오류 이벤트

JSONL append 시에도 한 줄을 완성한 뒤 flush한다. 마지막 줄이 비정상인 경우 이전 정상 줄까지 읽고 손상 줄을 별도 보고한다.

핵심 스캔 원본과 의사결정 스냅샷은 JSONL이 아니라 **파일 하나가 완결된 JSON**으로 저장한다.

## 4. JSON 파일별 데이터 계약

## 4.1 `schema.json`

전체 로컬 원본 포맷을 설명한다.

```json
{
  "schema_version": 1,
  "format": "radar-local-json",
  "authority": "local_raw_json",
  "scope": "observed_union_not_full_market",
  "timezone": "Asia/Seoul",
  "immutable_scan_files": true
}
```

## 4.2 `models/rank4-v1.json`

정렬 모델 버전과 prior 근거를 저장한다.

```json
{
  "schema_version": 1,
  "model_version": "rank4-v1",
  "policy_name": "rank4",
  "source_commit": "7f3acab",
  "effective_from": "20260713",
  "effective_at": "2026-07-13 09:00:00 KST",
  "prior": {
    "source": "chairman_40y_rule",
    "strength": "strong",
    "summary": "회장님 경험칙과 정렬4 합의"
  }
}
```

정렬4 목표 발효일은 2026-07-13이다. 그 전에 구현·검증·배포가 완료되지 않으면 실제 배포 완료 다음 거래일로 미룬다. 코드 배포보다 이른 effective date는 금지한다.

2026-07-10 데이터는 `mixed-deployment`로 기록하고 `rank4-v1 forward`에서는 제외한다.

## 5. 순위와 forward 데이터 계약

## 5.1 모델 버전

모든 scan·decision·trade·evaluation JSON에 다음을 저장한다.

- `rank_policy_name`
- `rank_model_version`
- `rank_model_effective_from`
- `rank_model_source_commit`

## 5.2 진짜 forward 조건

1. `signal_date`가 실제 effective date 이상
2. 신호 당시 모델 버전 저장
3. 신호 당시 rank bucket 저장
4. 해당 의사결정 모집단에 실제 포함
5. 익일 평가 결과 존재

7월 10일 서산은 `mixed-deployment` reference로 보존하되 forward에서 제외한다.

## 5.3 순위 시점

다음 순위를 구분한다.

- `precut_rank`
- `published_rank`
- `first_seen_rank`
- `latest_published_rank`
- `krx_decision_rank`
- `nxt_decision_rank`
- `eod_rank`

기존 `final`은 하위호환을 위해 유지하되 새 성과 계산의 단일 기준으로 사용하지 않는다.

## 5.4 prior·retro·forward

Git 파생 performance JSON에서 다음을 분리한다.

- `rank_prior`
- `rank_bucket_stats_retro`
- `rank_bucket_stats_forward`
- `rank_eval`

forward 킬스위치는 저장된 forward 표본만 사용한다. 과거 피처를 현 규칙으로 재분류한 retro 표본을 읽지 않는다.

소표본은 숨기지 않는다.

- n=1도 값 표시
- 고유종목 수 표시
- Wilson 하한 표시
- 평균·중앙·최저 고가 분리
- 표본 부족은 `수집 중` 라벨
- 자동 제외 금지

## 7. 기록 실패와 자동매매 안전 경계

## 7.1 주문을 막지 않는 선택적 로컬 기록

- scan 원본 JSON
- 분봉 JSON
- 연구용 rank event
- prior/retro/forward 표시 메타
- checksum·manifest 갱신

이 저장 실패는 로그와 경고를 남기되 기존 주문 종목·수량·안전필터를 변경하지 않는다.

## 7.2 반드시 fail-closed인 핵심 상태

- 레이더 파일 로드 실패 또는 stale
- 기존 포지션 원장 로드 실패
- 중복매수 판정 불가
- 예산·수량·안전필터 계산 실패

- 수동 확인 필요

1. SQL·SQLite 코드와 파일이 추가되지 않았다.
2. Mac 회차별 JSON이 권위 원본으로 동작한다.
3. 로컬 원본에 용량 제한·자동 축약·자동 삭제가 없다.
4. `data/local/`이 Git에서 제외된다.
5. secret이 JSON에 저장되지 않는다.
6. 회차 JSON은 원자적으로 저장되고 완성 후 불변이다.
7. 분봉 JSON은 원자 병합된다.
8. manifest는 원본 파일에서 재생성 가능하다.
9. rank_bucket 순서와 자동매매 정책이 바뀌지 않았다.
10. rank4-v1 effective date가 실제 배포 이후다.
11. 7월 10일 mixed-deployment가 forward에서 제외된다.
12. prior/retro/forward가 분리된다.
13. n=1도 저장·표시된다.
14. forward 킬스위치는 forward만 읽는다.
15. precut·published·KRX·NXT·EOD 순위가 복원된다.
16. 자동매매가 읽은 레이더 시각·모델·순위가 복원된다.
17. 전체 관찰 후보의 게이트 결과와 탈락 사유가 저장된다.
18. 분봉 원본과 스파크 결과가 저장된다.
19. 데이타솔루션 유형 비선정 종목을 재구성할 수 있다.
20. API 오류와 조건 탈락이 구분된다.
21. 모델 버전별 top1/top3 성과가 계산된다.
22. 선택적 JSON 저장 실패가 기존 주문 로직을 바꾸지 않는다.

24. 기존 history가 원자적으로 저장된다.
25. publish가 전역 순서를 깨뜨리지 않는다.
26. 기존 웹 JSON과 하위호환된다.
27. 모든 테스트·dry-run·웹 build가 통과한다.
28. 자동 백업·삭제·아카이브가 추가되지 않았다.

## 13. 최종 확정 문장

회장님의 40년 경험칙과 현재 rank_bucket 뼈대는 그대로 유지한다. 소표본은 하나도 버리지 않는다.

레이더가 실제 관찰한 모든 후보, 게이트 결과, 탈락 사유, 분봉 원본, 순위 변화, KRX·NXT 의사결정 스냅샷과 익일 결과를 **Mac 로컬의 날짜별·회차별 JSON 파일에 전량 보존한다.** SQL과 SQLite는 사용하지 않는다.

Git에는 웹과 검증에 필요한 compact 파생 JSON만 저장한다. Mac 원본 JSON에는 용량 제한, 자동 축약, 자동 삭제, 자동 아카이브를 적용하지 않는다. 백업은 회장님이 수동으로 수행한다.

실제 배포 이후 `rank4-v1` 표본부터 진짜 forward로 인정하고 prior·retro·forward를 분리해 모델 버전별 익일 최고 고가 성과를 계산한다. 실제 정렬 변경은 회장님께 성과를 보고하고 승인받은 뒤 새 모델 버전에서만 수행한다.
