# 문제3 - 2회 재분석 최종 의견서

작성일: 2026-07-10  
분석 대상: `문제1.md`, `문제2.md`, 정렬 합의 문서, 레이더 전체 순위·저장·백테스트·화면 코드, 현재 저장 데이터

## 0. 최우선 전제

회장님의 지시를 이번 최종 판단의 전제로 고정한다.

> 소표본이라는 이유만으로 무시하지 않는다. 현재 알고리즘의 뼈대는 회장님의 40년 시장 경험에서 나온 선행 지식이며, 단기 표본만으로 그 뼈대를 폐기하거나 뒤집지 않는다.

이 전제는 통계를 버리라는 뜻이 아니다. 올바른 해석은 다음과 같다.

1. 회장님의 경험칙은 단순 추측이 아니라 강한 사전지식(prior)으로 취급한다.
2. 소표본도 관측되는 즉시 기록하고 방향성 증거와 킬스위치 증거로 사용한다.
3. 표본이 작다는 이유로 결과를 숨기거나 정렬에서 자동 제외하지 않는다.
4. 반대로 소표본의 명목 승률만으로 회장님의 뼈대를 매일 뒤집지도 않는다.
5. 경험칙, 소급 실측, 도입 후 전진 실측을 서로 다른 필드와 표로 분리한다.
6. 최종 승격·강등은 사전에 정한 조건을 근거로 회장님이 결정한다.

이번 재분석의 핵심은 **소표본을 버리지 않으면서도 서로 다른 근거를 섞어 거짓 숫자를 만들지 않는 구조**를 확정하는 것이다.

## 2. 1차 재분석 결과

### 2.1 문제2의 9개 사실 판정은 모두 유지한다

`문제2.md`가 확인한 다음 사실은 코드와 데이터에서 다시 재현됐다.

1. 버킷 번호와 현재 상호배타 실측 성과가 단조롭게 일치하지 않는다.
2. 신호일 후보 간 `top1_hit` 등 상대순위 지표가 없다.
3. 카드 기대치가 현재 상호배타 버킷 실측과 다르다.
4. 게시컷 밖 후보와 게이트 탈락 후보가 충분히 저장되지 않는다.
5. 같은 종목의 장중 회차별 순위가 마지막 값으로 덮어써진다.
6. 현재 버킷 성과는 과거 피처를 현 규칙으로 다시 분류한 소급 성과다.
7. publish의 regular/reaccum 재조립은 향후 전역 순서를 깨뜨릴 수 있다.
8. 버킷 성과에 final과 dropout이 섞여 있다.
9. history 쓰기는 원자적 교체 방식이 아니다.

따라서 `문제1.md`와 `문제2.md`의 사실 진단은 폐기할 것이 없다.

### 2.2 현재 정렬을 즉시 폐기하면 안 된다는 클로드 의견에 동의한다

현재 버킷 순서는 회장님의 경험칙, 과거 전수조사, 기존 실측을 함께 반영해 7월 10일 확정된 뼈대다.

지금 관측된 B3/B4, B6/B7 등의 역전만 보고 즉시 순서를 바꾸면 다음 문제가 생긴다.

- 새 정렬 도입 직후라 실제 forward 표본이 아직 없다.
- 규칙 변경 때마다 전진검증 시작점이 다시 움직인다.
- 회장님의 경험칙을 단기 관측치가 대체하게 된다.
- 사전에 고정한 킬스위치의 의미가 사라진다.

그러므로 **현재 rank_bucket 순서는 우선 유지**하는 것이 맞다.

### 2.3 단, 클로드의 "통계적으로 유의하지 않다"는 표현은 최종 판단 근거로 부적절하다

표본이 작고 구간이 겹친다는 사실은 맞다. 그러나 이것을 다음처럼 해석하면 안 된다.

```text
표본이 작다
-> 유의하지 않다
-> 관측 결과는 의미가 없다
-> 아무 조치도 하지 않는다
```

회장님의 알고리즘이 prior로 존재하는 현재 상황에서는 다음처럼 해석해야 한다.

```text
회장님 경험칙을 현 순위 prior로 유지
+ 작은 표본도 전량 기록
+ 사전 고정 킬스위치에 즉시 투입
+ 방향성이 반복되면 상신 강도를 높임
+ 회장님 승인 전 자동 재정렬 금지
```

즉 소표본 역전은 **정렬 즉시 변경의 충분조건은 아니지만 절대로 무시할 자료도 아니다.**

### 2.4 기대치 괴리의 원인이 모집단 정의 불일치라는 클로드 정정은 정확하다

`RANK_BUCKET_BASELINES`의 많은 수치는 겹침을 허용한 조건 코호트 성과다. 하지만 실제 `rank_bucket_info()`는 가장 낮은 버킷 하나만 적용하는 상호배타 분류다.

대표 차이:

| 버킷 | 코드 기준값 n/+7/평균고가 | 상호배타 현재값 n/+7/평균고가 |
|---:|---|---|
| B3 | 21 / 81.0% / +15.64% | 6 / 66.7% / +12.25% |
| B5 | 12 / 83.3% / +17.39% | 2 / 50.0% / +9.50% |
| B6 | 15 / 80.0% / +16.14% | 6 / 66.7% / +13.72% |
| B10 | 12 / 66.7% / +15.35% | 5 / 20.0% / +4.08% |
| B11 | 179 / 48.6% / +9.90% | 98 / 27.6% / +5.85% |

따라서 이것은 단순한 일일 갱신 지연이 아니다. 카드의 `expected_*`가 실제 배정 버킷의 상호배타 성과라고 읽히지만, 일부 값은 더 넓은 겹침 코호트의 수치다.

### 2.5 데이터 결과 저장의 기본 골격은 정상이다

확인된 history 상태:

- 일별 파일 26개
- 전체 레코드 187개
- 평가 완료 182개
- 정상 익일 결과 179개
- `final=True` 116개
- 정상 결과 179개 모두 익일 시가·고가·저가·종가 파생값 보유

다음 구조는 유지 가치가 있다.

- 신호일 공식 종가로 entry 재정합
- 익일 공식 일봉으로 결과 평가
- `final`로 마감 카드 잔존 여부 분리
- `next_high_pct` 등 고가 성과 보존
- 화면은 백엔드 배열순 그대로 렌더링

문제는 결과 저장 자체가 아니라 **비선정 후보와 순위 생성 과정의 저장 부족**이다.

## 3. 2차 재분석에서 추가로 확인한 문제

## 추가 문제 A. 현재 킬스위치는 forward 전용이 아니다

심각도: 최상

`radar_backtest.py`의 `rank_bucket_stats()`는 모든 과거 표본을 현재 `rank_bucket_info()`로 재분류한다. `_kill_switch_rows()`도 이 소급 재분류 집합을 그대로 사용한다.

따라서 현재 화면의 킬스위치 상태는 다음을 구분하지 못한다.

- 7월 10일 정렬4 도입 이전의 소급 표본
- 7월 10일 이후 실제 해당 버킷으로 게시된 forward 표본
- 당시 화면에는 다른 순위였던 과거 표본
- final 종목과 dropout 종목

특히 정렬4 합의의 bucket 1 조건은 "앞으로 발생하는 신규 표본 중 +7% 미달 1건"이다. 현재 코드는 전체 소급 집합의 최저 고가를 보기 때문에 결과적으로 실패를 잡을 수는 있어도, 그것이 신규 forward 실패인지 명시적으로 판정하지 않는다.

### 최종 판단

클로드가 P1로 둔 `rank_model_version`과 retro/forward 분리는 맞으며, 단순 개선이 아니라 **킬스위치의 계약을 지키기 위한 선행 필수 작업**이다.

필요 필드:

- `rank_model_version`
- `rank_policy_name`
- `rank_bucket_at_signal`
- `rank_at_signal`
- `rank_basis_at_signal`
- `ranked_at`

필요 통계:

- `rank_bucket_stats_retro`
- `rank_bucket_stats_forward`
- `kill_switches_forward`
- `kill_switches_retro_reference`

forward 킬스위치는 저장된 신호 당시 버킷만 읽어야 한다. 과거 피처를 현재 규칙에 넣어 재분류하면 안 된다.

## 추가 문제 B. expected 수치의 역할이 세 가지로 섞여 있다

심각도: 높음

현재 하나의 `expected_touch7_rate`와 `expected_high_pct` 안에 다음 근거가 섞일 수 있다.

1. 회장님의 경험칙 또는 전수조사 prior
2. 과거 179건 소급 코호트 실측
3. 정렬4 도입 이후 forward 실측

B0은 특히 코드 주석상 14만 전수 prior의 72%와 forward 2/2의 평균 고가 수치가 한 스냅샷 안에 같이 들어간다. 다른 버킷은 겹침 코호트 수치를 쓰면서 화면에서는 실제 버킷의 "과거 +7% 터치"처럼 보인다.

### 최종 판단

클로드의 "상호배타 소급 실측으로 expected를 교체"는 표시 오류를 줄이지만, 그대로 교체하면 회장님의 prior가 다시 숫자 뒤로 사라진다. 최종 구조는 **교체가 아니라 분리**여야 한다.

권장 구조:

```json
{
  "rank_prior": {
    "source": "chairman_40y_rule | census_140k | agreed_rule",
    "strength": "strong",
    "summary": "왜 이 버킷이 위에 있는가"
  },
  "retro_stats": {
    "population": "exclusive_all | exclusive_final",
    "n": 0,
    "unique_n": 0,
    "touch7_rate": null,
    "avg_high_pct": null,
    "median_high_pct": null
  },
  "forward_stats": {
    "model_version": "rank4-v1",
    "n": 0,
    "unique_n": 0,
    "touch7_rate": null,
    "avg_high_pct": null,
    "median_high_pct": null
  }
}
```

정렬은 prior를 유지할 수 있다. 다만 화면의 실측 수치는 어떤 모집단인지 명시해야 한다.

## 추가 문제 C. expected_high_pct의 문서·코드·화면 의미가 다르다

심각도: 높음

`정렬4.md`는 `expected_high_pct`를 "과거 중앙값 익일 고가"라고 정의한다. 그러나:

- `radar.py` 주석은 평균이라고 한다.
- 하드코딩된 값은 평균 고가 값이다.
- `publish.py` 주석도 평균이라고 한다.
- `SuspectCard.tsx` 화면도 "평균고가"라고 표시한다.

즉 SSOT 문서와 실제 구현의 통계량이 다르다.

### 영향

고가 데이터는 극단값의 영향을 많이 받는다. 평균과 중앙값은 의미가 다르므로 하나의 필드로 혼용하면 기대치를 과대 표시할 수 있다.

### 최종 판단

하나를 선택해 덮어쓰지 말고 둘 다 저장해야 한다.

- `avg_next_high_pct`
- `median_next_high_pct`
- `min_next_high_pct`
- 필요하면 `p25_next_high_pct`

화면에서 "평균"과 "중앙"을 정확히 구분해야 한다.

## 추가 문제 D. rank_bucket_stats_snapshot에 합의된 핵심 근거가 빠져 있다

`정렬4.md`는 판정 당시 스냅샷에 `n`과 Wilson 하한을 저장하도록 요구한다. 그러나 현재 `RANK_BUCKET_BASELINES`에는 대부분 다음만 있다.

`wilson7_lower`, 중앙고가, 최저고가 등이 없다. 따라서 history에 `rank_bucket_stats_snapshot`을 저장해도 당시 보수적 근거를 완전히 재현할 수 없다.

### 최종 판단

스냅샷은 최소 다음을 가져야 한다.

- 근거 종류: prior/retro/forward
- 모집단: all/final/dropout
- 모델 버전
- 기준일과 평가 기간
- n, unique_n
- touch7/touch13
- Wilson 하한
- 평균/중앙/최저 고가
- 평균 종가, 상승마감률

## 추가 문제 E. 성과 화면이 valid=false를 실제로 구분 표시하지 않는다

심각도: 중간

백테스트 셀에는 `valid`가 있고 n이 기준 미만이면 false다. 그러나 현재 `RankBucketStatsPanel`은 `valid`를 읽어 "수집 중"으로 표시하지 않는다. 표본이 작은 셀도 큰 셀과 똑같이 색상과 퍼센트로 표시한다.

이 문제의 올바른 해결은 소표본 수치를 숨기는 것이 아니다.

### 올바른 표시

- 관측값은 그대로 표시한다.
- `n`, 고유종목 수를 같이 표시한다.
- prior와 관측값을 같이 표시한다.
- `forward n 부족`, `관찰 중` 상태를 명시한다.
- 소표본이라고 회색으로 지우거나 순위에서 자동 제외하지 않는다.

즉 **소표본 존중과 소표본 출처 표시는 동시에 가능**하다.

## 추가 문제 F. final과 dropout의 성과 차이가 실제로 크다

심각도: 높음

2차 재계산 결과 일부 버킷은 final과 dropout의 성과 차이가 크다.

| 버킷 | final n / +7 / 평균고가 / 종가평균 | dropout n / +7 / 평균고가 / 종가평균 |
|---:|---|---|
| B3 | 3 / 33.3% / +6.68% / -5.60% | 3 / 100% / +17.82% / +8.68% |
| B6 | 4 / 75.0% / +17.59% / +4.24% | 2 / 50.0% / +5.98% / -1.42% |
| B7 | 17 / 82.4% / +17.09% / +1.45% | 8 / 50.0% / +9.13% / -4.37% |

### 최종 판단

## 5. 최종 수정 우선순위

## P0. 현재 정렬 뼈대 동결

- 현재 rank_bucket 순서를 즉시 바꾸지 않는다.
- 회장님의 40년 경험칙을 `rank_prior`로 명문화한다.
- 자동매매 비중과 rank 정책은 별도 승인 전 변경하지 않는다.
- 소표본을 이유로 버킷을 삭제하지 않는다.

## P1. 모델 버전 및 진짜 forward 기록

- `rank_model_version` 도입
- 신호 당시 버킷과 순위를 영구 저장
- retro와 forward 표 완전 분리
- forward 킬스위치 별도 계산
- 7월 10일 이후 실제 저장값만 forward로 인정

이 작업이 완료되지 않으면 7월말 채점도 다시 소급 성과와 섞인다.

## P2. 전체 후보·탈락 사유·컷 전 순위 저장

클로드 문서에서는 P7이었지만 최종안에서는 P2로 올린다.

이유:

- 데이타솔루션 같은 강한 비선정 종목을 복원해야 한다.
- 회장님 알고리즘의 각 게이트가 실제로 유효한지 검증해야 한다.
- false negative는 선택된 suspects만 저장해서는 절대 찾을 수 없다.
- 분봉은 이후 소급이 어려우므로 오늘 저장하지 않으면 영구 소실된다.

1차 구현은 저장량을 통제한 축소판으로 시작해도 된다.

- 일마감 전체 추적 후보 스냅샷
- 컷 전 순위
- 게시 여부
- 게이트별 통과 여부
- 탈락 단계와 복수 탈락 사유
- 분봉 스파크 요약
- 데이터 조회 실패와 진짜 조건 탈락 구분

## P3. prior/retro/forward 기대치 분리

- `expected_*` 단일 필드 의존 제거
- prior 근거를 문자열과 버전으로 보존
- 상호배타 retro all/final/dropout 분리
- forward 통계 별도 저장
- 평균·중앙·최저 고가 분리
- Wilson 하한과 고유종목 수 포함

화면에는 최소한 다음이 보여야 한다.

```text
정렬 근거: 회장님 경험칙 + 급소/회전150
과거 소급: n=5, +7% 100%, 중앙고가 ...
정렬4 이후: n=0, 수집 중
```

소표본은 숨기지 않는다.

## P4. 상대순위 평가 추가

- 날짜별 실제 익일 고가 winner 생성
- top1/top3 적중 측정
- 순위 상관 측정
- 모델 버전별 분리
- 후보 수가 1개인 날은 top1 통계에서 별도 분리

## P5. final/intraday/dropout 모집단 분리

- 카드 expected는 사용 시점에 맞는 모집단 사용
- 마감 카드에는 final 통계를 우선 표시
- 장중 알림 평가는 intraday 통계 사용
- dropout은 필터 유효성 평가용으로 유지

## P6. 회차별 rank_path 저장

- 전체 원시 분봉을 매번 저장할 필요는 없다.
- 순위나 핵심 피처가 변할 때만 이벤트 append 가능
- `first_seen`, `first_top3_at`, `last_top3_at`, `rank_path` 저장
- 사용자가 본 장중 순위와 최종 순위를 모두 평가

## P7. 저비용 안정성 결함 수정

- history 임시파일 + fsync + `os.replace`
- publish 재조립 시 원 배열 순서 유지
- 전후 순서 회귀 테스트
- `expected_high_pct` 평균/중앙 의미 정정
- `rank_bucket_stats_snapshot` 필수 통계 완전 저장
- `valid=false`여도 수치는 표시하되 `forward 수집 중` 상태 병기

P7은 우선번호가 뒤일 뿐, 구현 비용이 작으므로 P1 작업과 함께 먼저 적용해도 된다.

## 7. 완료 검증 기준

향후 수정이 제대로 됐다고 인정하려면 다음을 모두 만족해야 한다.

1. 현재 rank_bucket 순서는 승인 없이 변하지 않는다.
2. 새 history 레코드에 모델 버전과 신호 당시 버킷이 저장된다.
3. forward 통계는 저장된 버킷만 읽는다.
4. retro 통계는 별도 표에서만 현재 규칙으로 재분류한다.
5. forward 킬스위치가 retro 표본을 읽지 않는다.
6. 전체 추적 후보에 탈락 사유가 저장된다.
7. 데이타솔루션 유형의 비선정 고회전 종목이 일마감 연구 데이터에 남는다.
8. final/intraday/dropout 통계가 분리된다.
9. 카드의 평균·중앙·prior·forward 문구가 실제 필드와 일치한다.
10. 소표본 수치는 숨기지 않고 n과 근거 출처를 함께 보여준다.
11. 날짜별 top1/top3 상대순위 성과가 모델 버전별로 계산된다.
12. history 파일이 원자적으로 교체된다.
13. publish 전후 suspects 순서가 의도 없이 변하지 않는다.
14. 기존 py_compile, rank bucket 테스트, publish dry-run, 웹 build가 통과한다.

## 8. 최종 의견

`문제1.md`의 문제 진단과 `문제2.md`의 사실 검증은 대부분 정확하다. 클로드가 제시한 가장 중요한 판단, 즉 **지금 정렬을 즉시 갈아엎지 말고 측정 구조부터 고쳐야 한다**는 결론에도 동의한다.

그러나 소표본에 대한 최종 해석은 수정해야 한다.

> 소표본은 무시할 이유가 아니다. 회장님의 40년 경험칙이 알고리즘의 강한 prior이므로 현재 정렬 뼈대는 유지한다. 동시에 n=1부터 모든 성공·실패를 저장해 사전 고정 킬스위치와 전진검증에 즉시 사용한다. 작은 표본은 자동 재정렬의 근거가 아니라, 회장님께 승격·강등을 상신하는 누적 증거다.

현재 가장 큰 문제는 정렬 순서 그 자체보다 **근거의 종류가 섞여 있고, forward라고 부르는 표가 실제 forward가 아니며, 탈락 후보가 저장되지 않는 것**이다.

따라서 최종 실행 원칙은 다음 한 문장으로 정리한다.

> **회장님의 정렬 뼈대는 유지한다. 소표본은 하나도 버리지 않는다. prior·소급·전진 데이터를 분리하고, 전체 후보와 탈락 사유를 저장한 뒤, 모델 버전별 상대순위 성과와 킬스위치를 근거로 회장님 승인 아래 정렬을 조정한다.**
