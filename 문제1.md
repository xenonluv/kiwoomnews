# 레이더 메인 suspects 순위 및 데이터 저장 구조 문제 분석

작성일: 2026-07-10  
분석 대상: `scripts/radar.py`, `scripts/publish.py`, `scripts/radar_backtest.py`, `web/data/radar.json`, `web/data/performance.json`, `data/radar_history/*.json`, 메인 레이더 프론트엔드

## 1. 분석 목적

레이더 메인 첫 페이지의 `suspects` 종목은 다음 거래일에 가장 높은 고가 상승률을 달성할 가능성이 높은 종목부터 나열되어야 한다.

이를 만족하려면 최소한 다음 조건이 성립해야 한다.

1. 스캐너가 발견한 비교 대상 종목과 피처가 누락 없이 저장되어야 한다.
2. 각 종목의 신호일 종가와 익일 시가·고가·저가·종가가 정확히 연결되어야 한다.
3. 실제 정렬키가 익일 고가 확률 또는 검증된 예상 고가와 단조롭게 일치해야 한다.
4. 화면 순위, 저장 순위, 백테스트 순위가 같은 의미를 가져야 한다.
5. 새 정렬 규칙은 과거 소급 분류와 실제 전진검증을 구분해야 한다.

현재 코드는 게시 및 익일 평가의 기본 파이프라인은 동작하지만, 핵심 목표인 "익일 최고 고가 가능성 순" 정렬은 충족하지 못한다.

## 2. 현재 데이터 흐름

현재 흐름은 다음과 같다.

```text
radar.py
  후보 탐지
  -> rank_bucket 계산
  -> _rank_sort_key 정렬

publish.py
  -> 게시 상한만큼 후보 절단
  -> data/radar_history/YYYYMMDD.json 기록
  -> web/data/radar.json 게시

메인 프론트엔드
  -> radar.json의 suspects 배열순 그대로 렌더링

radar_backtest.py
  -> 익일 공식 일봉 연결
  -> next_open/high/low/close 및 수익률 계산
  -> web/data/performance.json 갱신
```

`radar.py`가 만든 최종 배열 순서를 프론트엔드가 임의로 바꾸지는 않는다. 따라서 화면 순서가 잘못됐다면 주된 원인은 프론트엔드가 아니라 백엔드 정렬 규칙과 데이터 모집단에 있다.

## 문제 1. 실제 정렬키가 익일 고가 확률이 아니다

심각도: 높음

### 현재 구현

`scripts/radar.py`의 `_rank_sort_key()`는 다음 순서로 정렬한다.

1. `rank_bucket` 오름차순
2. 매우좋음 Tier1/Tier2 순서
3. `suspicion_score` 내림차순
4. 흔들기 `fade_pct` 내림차순
5. `peak_turnover_pct` 내림차순
6. `turnover_2d_pct` 내림차순
7. 종목명

즉 실제 최우선 정렬키는 확률이나 예상 고가가 아니라 사람이 부여한 버킷 번호다.

### 실제 성과와의 불일치

`web/data/performance.json`에 저장된 현재 버킷별 성과는 다음과 같다.

| 버킷 | 표본 n | 익일 +7% 터치율 | 평균 익일 고가 |
|---:|---:|---:|---:|
| B0 | 2 | 100.0% | +29.97% |
| B1 | 5 | 100.0% | +27.42% |
| B2 | 14 | 85.7% | +16.07% |
| B3 | 6 | 66.7% | +12.25% |
| B4 | 8 | 87.5% | +16.22% |
| B5 | 2 | 50.0% | +9.50% |
| B6 | 6 | 66.7% | +13.72% |
| B7 | 25 | 72.0% | +14.54% |
| B8 | 8 | 75.0% | +11.04% |
| B10 | 5 | 20.0% | +4.08% |
| B11 | 98 | 27.6% | +5.85% |

버킷 번호가 낮을수록 실제 성과가 항상 높지 않다.

- B3이 B4보다 앞이지만 B4의 +7% 터치율과 평균 고가가 모두 높다.
- B6이 B7보다 앞이지만 B7의 +7% 터치율과 평균 고가가 모두 높다.
- B7보다 뒤인 B8의 +7% 터치율이 더 높다.

자동 비교에서도 평균 고가와 +7% 터치율 모두 버킷순 단조 감소가 성립하지 않았다.

### 영향

- 메인 1위가 실제로 가장 높은 익일 고가 가능성을 가진 종목이라고 말할 수 없다.
- 상위 3개를 사용하는 자동매매 후보도 같은 정렬 오류의 영향을 받는다.
- 사용자에게 표시하는 순위가 과거 실측 수치와 모순될 수 있다.

### 필요한 수정 방향

- 정렬 목표를 명시적으로 하나로 고정해야 한다.
  - 예: `P(익일 고가 +7% 이상)`
  - 예: 익일 고가 상승률의 보수적 기대값
  - 예: 같은 날 suspects 중 최고 고가 1위를 할 확률
- 버킷 번호 자체가 아니라 검증된 확률 또는 기대값을 정렬키로 사용해야 한다.
- 소표본은 단순 관측률 대신 Wilson 하한, 베이지안 수축값 또는 최소 표본 가드를 적용해야 한다.

## 문제 2. 현재 시스템은 "그날 최고 고가 종목" 확률을 계산하지 않는다

심각도: 높음

### 현재 평가 방식

현재 백테스트는 각 종목을 독립적으로 평가한다.

- 익일 종가 상승 여부
- 익일 고가 +3%, +7%, +11%, +13% 터치 여부
- 익일 평균 고가 상승률
- 익일 평균 종가 수익률

그러나 같은 신호일의 후보들을 서로 비교해서 다음을 계산하지 않는다.

- 그날 suspects 중 실제 익일 고가 상승률 1위 종목
- 화면 rank 1이 실제 고가 1위를 차지했는지
- rank 1, 2, 3별 최고 고가 승리 확률
- 후보 간 상대 순위 상관관계
- NDCG, Spearman 상관, top-k 적중률

### 영향

현재 성과표만으로는 "이 버킷은 평균적으로 잘 올랐다"는 판단은 가능하다. 하지만 "오늘 후보 가운데 이 종목이 내일 가장 높이 오른다"는 상대 순위 문제는 검증할 수 없다.

즉 현재 모델은 코호트 분류기이지, 후보 간 최고 고가 순위 모델은 아니다.

### 필요한 수정 방향

신호일 단위 평가를 추가해야 한다.

```text
신호일별 final suspects 수집
-> 각 종목 next_high_pct 계산
-> 실제 next_high_pct 내림차순 actual_rank 생성
-> 게시 rank와 actual_rank 비교
```

필수 지표:

- `top1_hit`: 화면 1위가 실제 익일 고가 1위인지
- `top3_contains_winner`: 실제 고가 1위가 화면 상위 3개에 포함되는지
- `rank_spearman`: 게시 순위와 실제 고가 순위의 상관
- `rank1_avg_high_pct`: 화면 1위의 평균 익일 고가
- `rank2_avg_high_pct`, `rank3_avg_high_pct`
- 동률 처리 규칙

## 문제 3. 동적 백테스트 결과가 실제 정렬에 반영되지 않는다

심각도: 높음

### 현재 구현

`scripts/radar.py`의 `RANK_BUCKET_BASELINES`에는 버킷별 표본 수, +7% 터치율, 예상 고가가 코드에 하드코딩되어 있다.

한편 `scripts/radar_backtest.py`는 매일 최신 결과를 집계해서 `web/data/performance.json`에 저장한다. 하지만 `radar.py`는 이 최신 성과 파일을 읽어서 정렬하거나 카드 예상값을 갱신하지 않는다.

### 확인된 괴리

대표적으로 B10은 다음처럼 차이가 크다.

| 항목 | 하드코딩 기준값 | 현재 performance 실측 |
|---|---:|---:|
| +7% 터치율 | 66.7% | 20.0% |
| 평균 익일 고가 | +15.35% | +4.08% |

따라서 카드에 저장되는 다음 필드가 최신 성과와 다를 수 있다.

- `expected_touch7_rate`
- `expected_high_pct`
- `rank_bucket_stats_snapshot`

### 영향

- 성과 대시보드는 최신인데 메인 정렬은 과거 고정값을 사용한다.
- 성과 악화가 확인돼도 순위가 자동으로 내려가지 않는다.
- 사용자에게 보이는 기대값과 실제 검증표가 서로 모순된다.

### 필요한 수정 방향

- 백테스트가 승인된 정렬용 calibration 파일을 별도로 출력해야 한다.
- `radar.py`는 표본 수와 유효성 기준을 통과한 calibration만 로드해야 한다.
- 파일 손상·결측 시에는 버전이 기록된 보수적 기본값으로 폴백해야 한다.
- calibration 생성일, 근거 표본 기간, 표본 수, 모델 버전을 카드와 history에 저장해야 한다.
- 자동 반영과 수동 승인 중 어느 정책을 쓸지 명확히 정해야 한다.

## 문제 4. 게시 상한 밖 후보와 조건 탈락 데이터가 저장되지 않는다

심각도: 높음

### 현재 구현

`scripts/publish.py`는 `radar.py` 결과를 받은 뒤 기본 최대 12개로 자른다. 그 후 잘린 `out`만 `record_history()`에 넘긴다.

따라서 다음 종목은 history에 남지 않을 수 있다.

- 한 번도 게시 상한 안에 들어오지 못한 13위 이하 후보
- 회전율이 강했지만 최종 패턴 게이트를 통과하지 못한 종목
- 현재 등락률 밴드에서 근소하게 탈락한 종목
- 분봉 스파크 수가 부족했던 종목
- 유동비율 결측으로 회전율 판정을 못 한 종목
- 스캔 유니버스에 있었지만 각 단계에서 탈락한 종목

### 데이타솔루션 사례

데이타솔루션은 다음 데이터가 확인된다.

- 2026-07-09 폭발 레지스트리 저장
  - 고가 +29.97%
  - 유통회전율 108.6%
  - 거래대금 429억
- 2026-07-10
  - 고가 +13.08%
  - 종가 -5.44%
  - 거래량 22,020,375주
  - 유통회전율 약 400%
  - 거래대금 1,735억
  - 14:30 이후 1.5% 이상 5분 양봉 0개

당일 신규 폭발의 고가 +22% 조건, 재매집 등락률 밴드 및 분봉 조건을 충족하지 못했으므로 최종 suspects에는 들어가지 않았다. 문제는 "왜 탈락했는지"와 "당일 회전율 400%"가 일별 검증 데이터에 저장되지 않는다는 점이다.

### 영향

- 선택된 종목만 남아 선택편향이 발생한다.
- 탈락 조건이 실제로 유효했는지 검증할 수 없다.
- false negative를 찾기 어렵다.
- 데이타솔루션 같은 강한 비선정 종목과 상위 suspects의 익일 성과를 비교할 수 없다.
- 새로운 피처나 임계값을 과거 데이터로 재검증할 수 없다.

### 필요한 수정 방향

게시용 데이터와 연구용 데이터를 분리해야 한다.

```text
web/data/radar.json
  사용자에게 보여줄 상위 후보만 저장

data/radar_scan_history/YYYYMMDD/HHMM.json 또는 일별 append 저장소
  스캔된 전체 후보
  모든 피처
  통과/탈락 단계
  탈락 사유
  당시 순위키
  데이터 소스 및 결측 여부
```

최소 저장 필드:

- `scan_at`
- `code`, `name`
- `source_universe`
- `price`, `change_pct`, `high_pct`
- `volume`, `value`, `float_ratio`, `turnover_pct`
- `peak_date`, `peak_turnover_pct`
- `reignition_count`, `geupso_count`, `low_accum_count`
- `shakeout`, `fade_pct`, `dd6_pct`, `peak_dd_pct`
- `gate_passed[]`
- `reject_stage`
- `reject_reasons[]`
- `precut_rank`, `published`

## 문제 5. 장중 순위 변화와 시점별 피처가 보존되지 않는다

심각도: 중간

### 현재 구현

`record_history()`는 같은 날짜와 종목코드에 대해 새 행을 추가하지 않고 기존 레코드를 덮어쓴다.

보존되는 주요 시점 정보는 다음 정도다.

- `first_seen`
- 마지막 회차의 `rank`
- 마지막 회차의 피처
- 마지막 회차의 `final`

### 잃어버리는 정보

- 10:00 당시 순위와 점수
- 11:00 당시 회전율과 등락률
- 14:30 이후 분봉 신호가 생기며 순위가 바뀐 과정
- 상위권 진입·이탈 시각
- 장중 1위가 마감에 사라진 사례
- 사용자가 실제 화면에서 봤던 각 시점의 상태

### 영향

종가 기준 최종 카드 전략은 어느 정도 평가할 수 있지만, 장중 메인 화면과 실시간 알림의 순위 품질은 평가할 수 없다.

### 필요한 수정 방향

- 일별 종목 최종 레코드와 별개로 회차별 스냅샷을 append 방식으로 저장해야 한다.
- 최소한 순위 또는 핵심 피처가 바뀔 때만 이벤트를 저장하는 방식으로 용량을 줄일 수 있다.
- `first_seen`, `last_seen`, `first_top3_at`, `last_top3_at`, `rank_path`를 요약 필드로 생성해야 한다.

## 문제 6. 현재 rank_bucket 성과는 완전한 전진검증이 아니다

심각도: 중간

### 저장 상태

확인 당시 `data/radar_history`에는 다음 데이터가 있었다.

- 일별 파일 26개
- 전체 종목 레코드 187개
- 평가 완료 182개
- 정상 익일 결과 179개
- `final=True` 레코드 116개
- 정상 결과 179개 모두 익일 시가·고가·저가·종가 관련 값 보유

그러나 과거 187개 레코드 중 `rank_bucket`이 실제 신호 당시 저장된 레코드는 1개뿐이었다. 새 버킷 체계 도입 전의 과거 레코드에는 당시 버킷이 없다.

### 현재 백테스트 방식

`radar_backtest.py`의 `_ranked_copy()`는 과거 피처를 현재 `rank_bucket_info()`에 다시 넣어 버킷을 소급 계산한다.

이 방식은 과거 피처로 현재 규칙을 연구하는 데는 유용하다. 그러나 다음 문제를 가진다.

- 규칙을 만든 데이터와 평가 데이터가 겹칠 수 있다.
- 과거 당시 실제 화면 순위와 현재 재분류 순위가 다르다.
- 하드코딩된 버킷 경계가 과거 결과를 보고 정해졌다면 데이터 누수가 발생한다.
- 소표본 버킷의 100% 성과가 실제 운영에서도 유지되는지 알 수 없다.

### 영향

현재 performance의 버킷 성과를 새 정렬의 실전 검증 결과로 그대로 해석하면 과대평가할 수 있다.

### 필요한 수정 방향

- `rank_model_version`과 `rank_bucket_at_signal`을 신호 시점에 영구 저장해야 한다.
- 소급 재분류 결과와 실제 당시 순위를 별도 필드로 분리해야 한다.
- 모델 버전 도입일 이후 표본만 forward 성과로 집계해야 한다.
- 최소 표본 수를 충족하기 전에는 실전 확률이 아니라 `관찰 중`으로 표시해야 한다.

---

## 문제 7. publish 단계가 향후 전역 정렬을 깨뜨릴 수 있다

심각도: 중간

### 현재 구현

`publish.py`는 정렬된 suspects를 다음 두 배열로 분리한다.

- `visible_experimental=False`인 regular
- `visible_experimental=True`인 reaccum

이후 두 배열을 다시 `regular + reaccum` 순서로 붙인다.

현재 파이프라인은 대부분 또는 전부 `visible_experimental=True`라 실제 장애가 드러나지 않는다. 하지만 regular 경로가 다시 활성화되면 `radar.py`에서 이미 계산한 전역 순위가 publish 단계에서 깨질 수 있다.

### 예시

```text
radar.py 원래 순서:
1. reaccum A
2. regular B
3. reaccum C

publish.py 재조립:
1. regular B
2. reaccum A
3. reaccum C
```

### 필요한 수정 방향

- 게시 상한만 적용한다면 원래 배열에서 순서를 유지한 채 선택해야 한다.
- 슬롯 정책이 꼭 필요하면 선택 후 `_rank_sort_key`로 다시 전역 정렬해야 한다.
- publish 전후 종목코드 순서가 동일한지 회귀 테스트를 추가해야 한다.

## 문제 8. 성과 통계의 모집단 의미가 메인 마감 순위와 섞여 있다

심각도: 중간

### 현재 구현

주요 종가 전략 통계는 `final=True` 표본을 사용한다. 이것은 마감 카드 기준으로 타당하다.

그러나 `rank_bucket_stats`는 core, experimental, dropout을 합친 전체 표본을 다시 분류한다. 즉 장중에는 잡혔지만 마감 카드에서 사라진 `final=False` 종목도 버킷 성과에 포함된다.

### 영향

- 메인 마감 카드에 표시되는 기대값과 버킷 성과의 모집단이 다를 수 있다.
- 종가 매수 기준의 익일 확률과 장중 포착 기준의 익일 확률이 섞인다.
- 사용자가 기대값을 어떤 진입 시점 기준으로 해석해야 하는지 불명확하다.

### 필요한 수정 방향

다음 통계를 분리해야 한다.

- `rank_bucket_stats_final`: 마감 카드 잔존 종목
- `rank_bucket_stats_intraday`: 장중 한 번이라도 포착된 종목
- `rank_bucket_stats_dropout`: 마감 전 탈락 종목

메인 마감 카드의 예상값은 `final` 모집단을 사용해야 한다.

## 문제 9. history 파일 쓰기가 원자적이지 않다

심각도: 낮음~중간

### 현재 구현

일별 history는 파일을 직접 열어 `json.dump()`로 덮어쓴다. 프로세스 강제 종료, 디스크 오류, 시스템 종료가 쓰기 도중 발생하면 JSON이 잘릴 수 있다.

손상 파일 발견 시 `.corrupt`로 백업하는 방어는 있지만, 이미 손상된 당일 이력을 자동 복구하지는 못한다.

### 필요한 수정 방향

다른 레지스트리 저장 방식처럼 다음 순서를 사용해야 한다.

```text
임시 파일 쓰기
-> flush/fsync
-> JSON 재검증
-> os.replace로 원자적 교체
```

## 3. 정상적으로 구현된 부분

문제가 없는 부분도 구분해야 한다.

### 3.1 백엔드 배열순과 메인 화면 순서는 일치한다

메인 `LiveRadar`는 `data.suspects.filter()`만 수행하고 별도 정렬하지 않는다. 필터는 남은 원소의 기존 순서를 유지하므로 `radar.json` 순서가 화면 카드 순서다.

### 3.2 현재 게시 데이터는 `_rank_sort_key` 순서와 일치한다

확인 당시 현재 `radar.json`의 suspects 배열은 `_rank_sort_key` 재계산 결과와 일치했다.

### 3.3 익일 일봉 연결 방식은 적절하다

백테스트는 장중 저장 가격을 그대로 확정 진입가로 사용하지 않고 신호일 공식 일봉 종가로 재정합한다. 이후 익일 공식 일봉을 사용해 다음 값을 계산한다.

- `next_open`
- `next_high`
- `next_low`
- `next_close`
- `next_open_pct`
- `next_high_pct`
- `next_low_pct`
- `return_pct`

확인된 정상 결과 179건은 위 익일 파생값이 모두 저장되어 있었다.

### 3.4 final 구분은 유용하다

장중에 포착됐으나 마감 카드에서 사라진 종목과 마감까지 남은 종목을 `final`로 구분한다. 종가 매수 전략을 평가할 때 필요한 필드다.

### 3.5 최소 버킷 회귀 테스트는 통과한다

`scripts/rank_bucket_smoke_test.py`는 버킷 판정 14개와 매우좋음 Tier 내부 순서를 확인하며 현재 통과한다. 다만 이 테스트는 구현이 문서화된 고정 버킷 규칙과 같은지만 확인할 뿐, 그 규칙이 실제 익일 고가순으로 올바른지는 검증하지 않는다.

## 4. 권장 수정 우선순위

### 1단계: 데이터 보존부터 수정

1. 전체 스캔 후보 및 탈락 사유 저장
2. 게시 전 순위와 게시 여부 저장
3. 회차별 순위 스냅샷 저장
4. `rank_model_version` 저장
5. history 원자적 쓰기 적용

데이터를 먼저 보존하지 않으면 이후 정렬 모델을 개선해도 false negative와 탈락 조건을 검증할 수 없다.

### 2단계: 순위 평가 지표 추가

1. 신호일별 실제 익일 고가 순위 생성
2. top1 적중률 계산
3. top3 winner 포함률 계산
4. rank별 평균·중앙 익일 고가 계산
5. Spearman 및 NDCG 계산
6. final/intraday/dropout 모집단 분리

### 3단계: 정렬 점수 교체

1. 버킷 번호 중심 정렬 제거 또는 보조키로 강등
2. 표본 수를 반영한 보수적 확률 산출
3. 최신 calibration 파일을 정렬에 연결
4. 기대값 및 확률이 단조롭게 순위와 일치하는지 검증
5. 성과 악화 시 kill switch가 실제 정렬을 바꾸도록 연결

### 4단계: 전진검증

1. 새 모델 버전을 고정
2. 도입일 이후 데이터만 forward 표본으로 별도 집계
3. 최소 표본 도달 전 자동매매 확대 금지
4. 기존 순위와 새 순위를 일정 기간 동시에 그림자 기록
5. top1/top3 성과가 기존 모델보다 실제로 개선됐을 때 승격

## 5. 최종 판단

현재 시스템은 다음 기능은 수행한다.

- suspects 탐지
- 고정 버킷 규칙에 따른 결정론적 정렬
- 게시된 배열순 그대로 메인 화면 렌더링
- 신호일 공식 종가와 익일 OHLC 연결
- 익일 고가 및 종가 성과 저장

그러나 다음 핵심 조건은 충족하지 못한다.

- 익일 가장 높은 고가를 달성할 확률 자체를 계산하지 않는다.
- 버킷 순서가 최신 실측 고가 성과순과 일치하지 않는다.
- 최신 성과가 실제 정렬에 자동 반영되지 않는다.
- 비선정·탈락 후보 데이터가 부족해 false negative를 검증할 수 없다.
- 장중 순위 이력이 없어 실시간 화면 순위의 품질을 평가할 수 없다.
- 현재 버킷 성과 상당 부분은 실제 전진검증이 아니라 과거 소급 재분류다.

따라서 현재 상태를 정확히 표현하면 다음과 같다.

> 게시 및 익일 결과 저장 파이프라인은 기본적으로 동작한다. 그러나 메인 suspects 순위는 검증된 익일 최고 고가 확률순 모델이 아니라, 과거 분석을 바탕으로 사람이 고정한 버킷 우선순위다. 데이터 저장 범위도 선택된 후보 중심이어서 현재 구조만으로는 이 순위가 시장에서 최선인지 증명할 수 없다.
