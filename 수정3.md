## 2. 수정 기간의 운영 안전 원칙

### 2.1 자동매매는 DRY로 유지한다

수정기에는 웹/KV 자동매매 토글을 OFF로 두고 cron에서도 `AUTOTRADE_LIVE=1`을 제거한다.

설치기 문제만 먼저 고쳤다고 즉시 `--live`로 재설치하지 않는다. 이슈 2의 전역 잠금과 이슈 1의 매수 체결확인을 모두 구현하고 다음 조건을 통과한 뒤에만 LIVE 전환을 검토한다.

- 전역 잠금 동시성 테스트 통과
- 매수 미체결·부분체결·전량체결 테스트 통과
- 제출 결과 불명 시 재주문 금지 테스트 통과
- 재시작 복구 테스트 통과
- 최소 1거래일 DRY 운영 관찰
- 실계좌·미체결 주문·포지션 원장 수동 대조
- 사용자의 명시적 LIVE 전환 승인

### 2.2 단계별 독립 커밋

각 이슈는 독립 커밋으로 처리한다. 한 커밋에서 순위 정책, 자동매매 수량, 연구 통계까지 함께 바꾸지 않는다.

각 단계의 테스트가 끝나기 전 다음 단계로 넘어가지 않는다. 실패하면 해당 커밋만 되돌릴 수 있어야 한다.

## 4. 이슈 3 — cron 설치기의 안전 기본 통일

### 4.1 권위 설치기

`scripts/install_cron_kiwoom.sh`를 유일한 SSOT로 정한다.

`scripts/install_cron.sh`는 자체 cron 문자열을 보유하지 않고 권위 설치기로 모든 인자를 전달하는 얇은 wrapper로 바꾼다.

### 4.2 확정 인터페이스

- 인자 없음: DRY 설치
- `--live`: LIVE 설치
- `--dry-run`: DRY 설정 미리보기
- `--live --dry-run`: LIVE 설정 미리보기
- `--uninstall`: 키움 cron 블록만 제거
- 알 수 없는 옵션: 비정상 종료

인자 없는 실행 결과에 `AUTOTRADE_LIVE=1`이 포함되면 실패다.

### 4.3 문서 통일

다음 문서의 설치 안내를 모두 같은 절차로 통일한다.

- `README.md`
- `CLAUDE.md`
- `HANDOFF.md`
- `업데이트mac.md`
- `7월말봐라.md`
- `맥설치매뉴얼.md`

문서의 표준 순서는 다음과 같다.

```text
기본 DRY 설치 → 테스트 → 계좌 대조 → 사용자 승인 → --live 설치 → KV 토글 ON
```

### 4.4 필수 검증

1. 인자 없는 dry-run 출력에 LIVE 환경변수가 없는지 확인
2. `--live --dry-run`에서만 LIVE 환경변수가 나타나는지 확인
3. 두 번 설치해도 프로젝트 cron 블록이 하나인지 확인
4. 다른 저장소 cron이 보존되는지 확인
5. 제거 명령이 키움 블록만 삭제하는지 확인

설치기 수정 직후에는 LIVE로 복원하지 않고 DRY 상태를 유지한다.

## 5. 이슈 2 — 자동매매 프로세스 전역 잠금

### 5.1 단일 실행 잠금

`/tmp/kiwoomnews-autotrade.lock`을 자동매매 전용 잠금 파일로 사용한다. publish의 git 잠금과 분리해 상호 대기를 만들지 않는다.

`autotrade_common.acquire_execution_lock()`을 만들고 Mac/Linux에서는 다음 방식으로 동작시킨다.

```text
fcntl.flock(LOCK_EX | LOCK_NB)
```

잠금 획득 실패 시 기다리지 않고 다음 로그를 남긴 뒤 주문 없이 종료한다.

```text
다른 자동매매 프로세스 실행 중 — 이번 회차 skip
```

### 5.2 공용 적용 대상

다음 진입점은 모두 같은 잠금을 사용한다.

- `autotrade_executor.run()`
- `autotrade_monitor.run()`
- 포지션을 변경하는 `autotrade_reconcile` 경로

잠금은 계좌조회, 주문, pending 저장, 포지션 저장이 끝날 때까지 유지한다. 모니터 API 호출이 길어져 다음 1분 회차가 skip되는 것은 정상 동작이다.

### 5.3 포지션 파일 강화

1. 임시파일명을 `.tmp.<pid>.<uuid>`로 만든다.
2. `dump → flush → fsync → 재파싱 → os.replace → 디렉터리 fsync` 순서로 저장한다.
3. 저장 전에 다음을 검증한다.
   - `qty_open >= 0`
   - closed 포지션의 `qty_open == 0`
   - 같은 주문번호의 pending 중복 없음
   - `accounted_filled <= ordered_qty`
4. 오류 시 기존 원장을 보존하고 예외를 상위로 전달한다.

### 5.4 필수 적대 테스트

1. 두 모니터를 동시에 시작해 매도 호출이 한 번인지 확인
2. 모니터와 실행기를 동시에 시작해 한쪽이 안전하게 skip되는지 확인
3. API를 70초 지연시켜 다음 cron 회차가 중복 주문 없이 종료되는지 확인
4. 프로세스 예외 종료 후 OS가 잠금을 회수하는지 확인
5. 서로 다른 종목을 동시에 처리하려 해도 원장 갱신이 유실되지 않는지 확인
6. 저장 실패 시 기존 JSON이 그대로인지 확인

이슈 2는 실주문 안전 경계이므로 구현 후 두 차례의 독립 코드 리뷰를 거친다.

## 6. 이슈 5 — 휴장일과 데이터 기준일 검증

### 6.1 기존 가드의 인정 범위

현재 `radar._allow_signal_date()`는 평일 09:00 이후 신호일이 오늘과 다르면 재매집·흔들기 신호를 차단한다. 따라서 휴장일 전일 자료로 suspects가 생성되는 최악의 일부 시나리오는 이미 막혀 있다.

그러나 다음은 남아 있다.

- 자동매매 모니터가 전일 가격으로 손절·익절을 계산할 가능성
- youtong/forecast의 stale 표시
- 연구 cohort의 거래일 오염
- 각 데이터 소스의 기준일 불일치

따라서 심각도는 `높음`으로 확정하고 중앙 검증을 추가한다.

### 6.2 메타데이터 반환

`minute_bars_today_with_meta()` 또는 호환 옵션을 추가해 다음 구조를 반환한다.

```python
{
    "trade_date": "YYYYMMDD",
    "bars": [],
    "fetch_status": "ok|empty|error|stale",
}
```

라이브 radar는 반드시 거래일 메타가 있는 경로를 사용한다. 일봉과 현재가도 확인 가능한 기준일을 함께 전달한다.

### 6.3 중앙 market state

요일만으로 거래일을 확정하지 않는다. 판정 우선순위는 다음과 같다.

1. 브로커 또는 거래소의 영업일·시장상태 응답
2. 당일 유효 일봉 존재와 응답 거래일 일치
3. 당일 분봉 거래일 일치
4. 로컬 휴장 캘린더는 보조 안전망

브로커 영업일 API가 없거나 불명확하면 `당일 일봉 존재 + 응답 거래일 일치`를 우선 사용한다. 서로 모순되면 열림으로 추정하지 않고 fail-closed 한다.

### 6.4 적용 경계

- radar: `STALE_TRADE_DATE`로 기록하고 `scan_ok=false`
- publish: 빈 레이더로 덮어쓰지 않고 기존 정상 파일 유지
- executor: 주문 직전 독립 검증, 불확실하면 매수하지 않음
- monitor: 가격 계산 전 독립 검증, 불확실하면 매도하지 않음
- research: 휴장일 cohort를 만들지 않음

### 6.5 필수 테스트

1. 평일 휴장일에 전일 분봉을 반환하는 응답
2. 정상 평일 09:01 당일 응답
3. 장전 당일 봉 미생성 상태
4. 일봉은 오늘, 분봉은 전일인 혼합 상태
5. 휴장일 publish가 기존 radar를 덮어쓰지 않는지 확인
6. 휴장일 매수·매도 함수 호출이 0건인지 확인
7. 연휴 후 첫 거래일 정상 복귀 확인

## 7. 이슈 1 — 매수 주문 접수와 체결 분리

### 7.1 절대 선행 조건: 매수 주문 조회 필드 확인

현재 `kt00007`의 매도측 필드는 테스트됐지만 매수측 필드는 검증되지 않았다. 필드명을 추정해서 구현하지 않는다.

확인 순서는 다음과 같다.

1. 키움 공식 문서와 기존 주문 응답에서 매수 조회 필드 확인
2. 시크릿을 제거한 실제 응답 fixture 확보
3. 문서만으로 불충분할 때에만 소액 실측 검토
4. 소액 실측은 사용자의 별도 승인을 받은 뒤 실시
5. 전역 잠금이 먼저 배포된 상태에서만 실측
6. 유동성 높은 종목, 최소 수량, 주문·취소 상태를 HTS로 동시 확인

실측 주문은 자동으로 실행하지 않는다. 승인 없이 주문을 내지 않는다.

### 7.2 공통 도구와 별도 상태 전이

매수와 매도에서 공통으로 사용하는 것은 다음 저수준 도구다.

- 주문번호 조회와 정규화
- 주문수량·체결수량·잔량 검증
- pending 원자 저장
- 취소·종결 확인
- 제출 결과 불명 처리

다만 상태 반영은 서로 다르므로 다음 함수를 분리한다.

- `reconcile_pending_entry()` — 확인된 체결분으로 포지션 생성·증가
- `reconcile_pending_exit()` — 확인된 체결분만 기존 포지션 감소·종료

매수와 매도에 하나의 동일 상태기계를 강제로 사용하지 않는다.

### 7.3 pending entry 구조

포지션 원장 루트를 하위호환 방식으로 확장한다.

```json
{
  "schema_version": 2,
  "positions": [],
  "pending_entries": []
}
```

상태는 다음과 같이 관리한다.

```text
PREPARED → ACCEPTED → PARTIAL | FILLED | CANCELLED | REJECTED
         ↘ SUBMIT_UNKNOWN
```

- `PREPARED`: 브로커 호출 전에 영구 저장
- `SUBMIT_UNKNOWN`: 접수 여부 불명, 자동 재주문 금지
- `ACCEPTED`: 주문번호 확보, 체결 미확정
- `PARTIAL`: 일부 체결, 잔량 pending 유지
- `FILLED`: 확인된 수량 전부 포지션 반영
- `CANCELLED/REJECTED`: 주문 종결

### 7.4 주문과 원장 반영 순서

1. 주문 intent를 `PREPARED`로 저장한다.
2. 저장 실패 시 브로커 주문을 호출하지 않는다.
3. 주문을 한 번만 호출한다. 자동 네트워크 재시도는 하지 않는다.
4. 주문번호를 확보하면 즉시 `ACCEPTED`로 저장한다.
5. 응답 불명 또는 주문번호 추출 실패 시 잠금을 유지한다.
6. 주문번호 조회의 `newly_filled`만 포지션에 반영한다.
7. 부분체결이면 실제 체결수량만 반영하고 잔량을 취소·재확인한다.
8. 평균 체결가가 검증돼야 손절·익절 기준가격으로 사용한다.
9. 체결가격을 확정할 수 없으면 `entry_price_verified=false`로 두고 자동 청산을 보류한다.
10. 미확정 pending이 있으면 동일 종목 재매수와 해당 예산 재사용을 차단한다.

### 7.5 예산과 중복매수 규칙

`deployed_today()`와 `already_bought()`는 체결 포지션뿐 아니라 pending intent도 인식해야 한다.

- pending 주문의 예약 예산은 가용예산에서 제외
- 같은 종목 pending 존재 시 신규 주문 금지

- 부분체결 시 체결금액과 미체결 예약금액을 구분

### 7.6 필수 적대 테스트

- 0~5%
- 5~10%
- 10~20%
- 20~40%
- 40~80%

- 150~300%
- 300% 이상

#### 사전등록 가설 뷰

4. 이 뷰를 운영 문턱으로 자동 적용하지 않는다.

- -5% 터치

## 10. LIVE 전환 승인 조건

다음 항목을 모두 충족하기 전에는 `--live`를 실행하지 않는다.

- [ ] 설치기 기본값이 DRY다.
- [ ] 실행기·모니터·reconcile이 동일 전역 잠금을 사용한다.
- [ ] 매수 주문 전에 pending intent가 영구 저장된다.
- [ ] 접수와 체결이 분리된다.
- [ ] 확인된 체결수량과 체결가만 포지션에 반영된다.
- [ ] SUBMIT_UNKNOWN 상태가 자동 재주문되지 않는다.
- [ ] 미체결·부분체결·전량체결·재시작 테스트가 통과한다.
- [ ] 휴장일에 게시·매수·매도가 발생하지 않는다.
- [ ] 최소 1거래일 DRY 관찰을 완료한다.
- [ ] 실계좌와 원장을 수동 대조한다.
- [ ] 사용자가 명시적으로 LIVE 전환을 승인한다.

LIVE 전환 후에도 처음에는 최소 수량으로 검증한다. 주문번호, 체결수량, 평균단가, pending 종결, 포지션 수량이 HTS와 모두 일치해야 정상으로 판정한다.

## 11. 즉시 롤백 조건

다음 중 하나라도 발생하면 KV 토글을 OFF로 하고 cron을 DRY로 다시 설치한다.

- 동일 intent 또는 동일 포지션의 중복 주문
- 원장수량과 실계좌수량 불일치
- 주문번호 없는 pending의 자동 해제
- 실제 체결수량보다 많은 포지션 기록
- 체결가 미확정 상태에서 자동 손절·익절
- 휴장일 또는 기준일 불일치 데이터로 신호·주문 생성
- 비게시 연구 통계가 운영 순위·가중치를 변경
- 포지션 원장 또는 manifest 손상

롤백 후에는 코드만 되돌리지 않는다. HTS의 미체결 주문과 잔고를 확인해 pending과 position 원장을 실제 계좌에 맞춘 뒤 재개한다.

## 12. 최종 합의 문장

> 설치기의 기본 LIVE 함정을 먼저 제거하고, 모든 자동매매 프로세스를 하나의 실행 잠금으로 직렬화한다. 휴장일과 데이터 기준일을 중앙에서 검증한 뒤, 공식 문서와 승인된 최소 주문으로 매수 조회 필드를 확정한다. 매수는 주문 전 pending을 영구 저장하고 실제 확인된 체결수량과 체결가만 포지션으로 전환한다. 비게시 종목의 익일 연구는 별도 작업과 별도 통계로 축적하며 운영 순위에는 자동 반영하지 않는다. 이 모든 검증과 최소 1거래일 DRY 관찰이 끝나고 사용자가 승인하기 전까지 자동매매 LIVE를 재활성화하지 않는다.

이 문장을 이슈 1~5 수정 작업의 최종 안전 기준으로 확정한다.
