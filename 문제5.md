## 0. 최종 원칙

이번 수정은 현재 순위 알고리즘을 교체하는 작업이 아니다.

1. 회장님의 40년 경험칙을 현재 정렬의 강한 prior로 유지한다.
2. 현행 `rank_bucket` 순서와 버킷 조건은 변경하지 않는다.
3. 소표본은 n=1부터 모두 저장하고 표시한다.
4. prior, 소급 재분류(retro), 도입 후 전진검증(forward)을 섞지 않는다.
5. 자동 판정은 상신 상태만 만들며 실제 승격·강등은 회장님 승인 후 적용한다.
6. 자동매매 종목 선택·비중·안전필터는 이번 작업에서 변경하지 않는다.
7. 데이터 조회 실패와 알고리즘 조건 탈락을 반드시 구분한다.
8. 새 정렬의 진짜 forward 시작일은 2026-07-13으로 고정한다.

최종 목표는 다음과 같다.

> 현 정렬 뼈대를 유지한 상태에서, 당시 실제 순위와 전체 후보·탈락 이유·실제 매매 의사결정 시점을 정확히 저장하고, 모델 버전별 익일 최고 고가 순위 성과와 forward 킬스위치를 왜곡 없이 계산한다.

## 1. 최종 코드리뷰에서 확인한 현재 구조

## 1.1 순위 생성

`scripts/radar.py`가 다음 순서로 처리한다.

```text
폭발 레지스트리 및 youtong 추적 풀 준비
```

정렬은 현재 정상적으로 결정론적이며, `web/data/radar.json`의 배열 순서와 메인 화면 순서도 일치한다.

## 1.2 게시와 history 저장

`scripts/publish.py`가 `radar.py` 결과를 받아 기본 12개로 제한한 뒤 다음을 수행한다.

- `data/radar_history/YYYYMMDD.json` 병합
- `web/data/radar.json` 게시
- 변경 시 git commit/push

같은 종목은 회차별 새 행을 만들지 않고 일별 한 레코드를 계속 덮어쓴다.

## 1.3 익일 평가

`scripts/radar_backtest.py`는 신호일 공식 일봉 종가를 entry로 재정합하고 익일 공식 일봉으로 다음을 저장한다.

- `next_high_pct`
- `next_low_pct`
- `return_pct`

이 결과 저장 골격은 정상이며 유지해야 한다.

## 2. 최종 리뷰에서 새로 확정된 핵심 결함

## 문제 A. 현재 `final`은 실제 KRX/NXT 의사결정 순위가 아니다

심각도: 최상

`publish.py`의 주석은 마지막 회차를 15:45라고 설명하지만 실제 운영 크론은 평일 09:01부터 20:51까지 10분 간격으로 실행된다.

자동매매 시점은 다음과 같다.

- KRX: 15:18
- NXT: 19:50

하지만 history의 동일 종목 레코드는 20:51까지 계속 덮어써진다. 현재 `final`은 마지막 게시 회차 잔존 여부에 가깝고 다음 중 어느 것도 정확히 의미하지 않는다.

- KRX 주문 직전 순위
- KRX 공식 마감 순위
- NXT 주문 직전 순위
- 사용자가 장중 처음 본 순위

### 영향

- KRX 자동매매가 실제로 읽은 순위와 history의 최종 순위가 다를 수 있다.
- NXT 주문 이후 20:51에 새로 들어온 종목이 `final=True`가 될 수 있다.
- 20:51 후보를 KRX 종가 매수 표본처럼 평가하면 시간상 불가능한 진입이 섞인다.
- 현재 final 성과는 실제 운용 시점 성과와 정확히 일치하지 않는다.

### 확정 수정

기존 `final`은 하위호환을 위해 당장 삭제하지 않는다. 대신 다음 의미가 명확한 필드를 추가한다.

- `first_seen_rank`: 최초 게시 순위
- `latest_published_rank`: 가장 최근 게시 순위
- `precut_rank`: 게시 상한 적용 전 전체 순위
- `krx_decision_rank`: 15:18 직전 마지막 게시 순위
- `nxt_decision_rank`: 19:50 직전 마지막 게시 순위
- `krx_decision_present`: KRX 의사결정 모집단 포함 여부
- `nxt_decision_present`: NXT 의사결정 모집단 포함 여부
- `eod_present`: 당일 마지막 운영 회차 잔존 여부
- `rank_path`: 회차별 순위 변화 요약

새 성과표는 `final` 하나를 사용하지 않고 목적별 모집단을 명시한다.

## 문제 B. 7월 10일 서산은 정렬4의 진짜 forward 표본이 아니다

심각도: 최상

정렬4 커밋 시각은 2026-07-10 20:06:55이다. 이는 KRX 마감과 NXT 자동매매 시점 이후다.

7월 10일 서산에는 `rank_bucket=4`가 저장됐지만, 신호 당일 실제 주문 가능한 시점에 정렬4가 운영되지 않았다. 따라서 이를 새 모델의 forward 실전 표본으로 세면 안 된다.

### 확정 수정

정렬 정책 메타데이터를 다음처럼 고정한다.

```json
{
  "rank_policy_name": "rank4",
  "rank_model_version": "rank4-v1",
  "rank_model_source_commit": "7f3acab",
  "rank_model_effective_from": "20260713",
  "rank_model_effective_at": "2026-07-13 09:00:00 KST"
}
```

forward 자격 조건:

1. `signal_date >= 20260713`
2. 신호 당시 `rank_model_version == rank4-v1`
3. 신호 당시 저장된 `rank_bucket_at_signal` 존재
4. 평가 결과 존재
5. 해당 성과표가 요구하는 시점 모집단에 포함

7월 10일 서산은 retro/reference에는 포함할 수 있지만 `rank4-v1 forward`에서는 제외한다.

## 문제 C. KRX 가격과 UN 거래량, NXT 등락률의 시점을 분리해야 한다

심각도: 높음

현재 설계는 다음 정책을 사용한다.

- 가격·OHLC: KRX 공식 J
- 거래량·거래대금: KRX+NXT 통합 UN
- 16:00 이후 현재 등락률 일부: NXT 재평가

이 정책 자체는 의도된 것이다. 문제는 저장 필드만 보면 어느 시점과 시장 기준인지 완전히 복원하기 어렵다는 점이다.

### 확정 수정

모든 의사결정·스캔 스냅샷에 다음을 저장한다.

- `observed_at`
- `trade_date`
- `price_basis`: `KRX`
- `money_basis`: `UN` 또는 `KRX`
- `change_basis`: `KRX` 또는 `NXT`
- `market_phase`: `preopen`, `krx_open`, `krx_close`, `nxt_after`, `operational_eod`
- `data_as_of`

KRX와 NXT 상태는 같은 필드를 덮어쓰지 않고 시점별 관측으로 보존한다.

## 3. 최종 데이터 모델

## 3.1 순위 정책 SSOT

새 파일 `scripts/rank_policy.py`를 만든다.

책임:

- `rank4-v1` 버전 상수
- effective date/time
- `RANK_BUCKET_BASELINES` 또는 prior 정의
- `rank_bucket_info()`
- `rank_shadow_buckets()`
- `_rank_sort_key()`에 필요한 정책 함수
- 모델 버전 메타데이터 반환

`radar.py`와 `radar_backtest.py`가 같은 모듈을 import한다.

### 이유

현재 `radar_backtest.py`가 큰 실행 모듈인 `radar.py`를 직접 import해 정렬 경계와 버킷 함수를 공유한다. 별도 정책 모듈로 분리하면 다음 장점이 있다.

- 순위 정책 버전이 명확해진다.
- backtest가 라이브 스캐너 전체에 결합되지 않는다.
- 버전별 회귀 테스트가 쉬워진다.
- 정렬 변경과 스캔 변경을 분리할 수 있다.

주의: 함수 이동은 동작 변경 없이 수행하고 기존 import 호환이 필요하면 `radar.py`에서 재노출한다.

## 4. 구현 단계

## 1단계. 순위 정책 버전과 저장 안전성

대상:

- `scripts/rank_policy.py` 신규
- `scripts/radar.py`
- `scripts/publish.py`
- `scripts/rank_bucket_smoke_test.py`

작업:

1. 현재 버킷 함수와 상수를 동작 변경 없이 정책 모듈로 분리한다.
2. `rank4-v1`, source commit, effective date를 정의한다.
3. 전체 정렬 직후 `precut_rank`를 부여한다.
4. publish 절단 후 원래 전역 순서를 보존하며 `published_rank`를 부여한다.
5. history 쓰기를 임시파일, flush, fsync, `os.replace`로 원자화한다.
6. regular/reaccum 슬롯 선택이 전역 순서를 깨지 않게 원 배열 기반 선택으로 바꾼다.
7. 순위 정책과 자동매매 정책은 변경하지 않는다.

검증:

- 기존 14개 버킷 스모크 테스트 통과
- 정책 모듈 이동 전후 동일 입력의 sort key 완전 일치
- current `radar.json` 순서 변화 없음
- regular/reaccum 혼합 배열 순서 회귀 테스트
- 손상/중단 쓰기 테스트에서 기존 history 보존

## 2단계. 의사결정 시점과 rank_path

대상:

- `scripts/publish.py`

## 10. 최종 확정 문장

현재 `rank_bucket` 정렬 뼈대는 회장님의 prior로 유지한다. 소표본은 하나도 폐기하지 않고 n=1부터 저장·표시한다. 2026-07-13부터 `rank4-v1`의 진짜 forward 표본을 시작하며, 과거 소급 재분류와 완전히 분리한다.

최종 수정의 첫 번째 목적은 정렬을 바꾸는 것이 아니라 **정렬이 실제 어느 시점에 어떤 종목을 몇 위로 보여줬고, 실제 자동매매가 어느 스냅샷을 읽었으며, 탈락한 종목은 왜 탈락했는지를 영구 복원 가능하게 만드는 것**이다.

그 데이터가 쌓인 뒤 모델 버전별 `top1_hit`, `top3_contains_winner`, 평균·중앙 익일 고가와 forward 킬스위치를 회장님께 보고한다. 실제 순위 승격·강등은 그 보고와 회장님 승인 이후 별도 버전에서만 수행한다.
