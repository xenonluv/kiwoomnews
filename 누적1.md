## 0. 한 줄 요약

## 2. 리스크 A — Vercel 재빌드 횟수 (일일 배포 한도)

### 2.1 메커니즘
`xenonluv/kiwoomnews`의 main에 push가 일어날 때마다 Vercel이 자동 재배포한다(Root Directory=`web`). publish cron은 **1,6,11,…,56분 = 시간당 12회 × 9~20시(12시간) = 이론상 하루 최대 144회**, 실제로는 "변경 시에만 push"라 거래일 ~278커밋 중 radar.json 변경분(자동게시 ~132)이 배포를 트리거한다. 회차이력 커밋도 push되면 함께 배포 트리거가 될 수 있다.

### 2.2 리스크
- **Vercel 무료(Hobby) 플랜은 Git 연동 배포에 일일 생성 한도(약 100회/일)와 월 빌드 실행시간 한도가 있다.** 거래일 배포 트리거가 100회를 넘어서면 배포가 큐잉·지연되거나 거부될 수 있다.
- 활발한 장세가 지속되면(수상종목이 매 회차 갱신) 트리거 빈도가 상시 상한에 근접한다.
- ⚠️ **미확인**: 실제 Vercel 대시보드의 일일 배포 수·빌드 시간·플랜 등급은 로컬에서 확인 불가. **회장님이 Vercel 대시보드에서 "지난 24h 배포 수"와 "이번 달 빌드 실행시간 사용률"을 한 번 확인**하시면 실제 여유가 명확해진다.

### 2.3 현재 상태
아직 배포 실패·한도 초과 징후는 로그에 없음(publish 로그 최근 에러 0건). **즉 지금은 정상, 상한 근접 여부만 확인 필요.**

### 2.4 완화 선택지 (필요 시, 지금 아님)
1. **push 빈도 축소** — 회차이력(`레이더 회차 이력 기록`) 커밋을 별도 브랜치/로컬 전용으로 돌리거나, radar.json 실질 변경이 있을 때만 push(이미 부분 적용). 회차이력을 Vercel 배포 대상에서 빼면 트리거가 ~절반으로.
2. **Vercel 빌드 스킵** — 데이터 전용 커밋에 `[skip ci]`/Ignored Build Step으로 radar.json만 바뀐 경우 프론트 재빌드를 건너뛰고 ISR/폴링으로만 갱신(프론트는 이미 60초 폴링·`/api/radar` 30초 캐시라 재빌드 없이도 최신 데이터 노출 가능).
3. Vercel 유료 전환(최후 수단).

> 특히 **선택지 2가 근본적**이다 — radar.json은 런타임에 API·폴링으로 읽히므로, 데이터만 바뀔 때 프론트 정적 재빌드는 사실상 불필요할 수 있다. (구현 전 SSG/ISR 의존성 확인 필요 — 별도 검토 과제.)

## 3. 리스크 B — git 히스토리 장기 팽창

### 3.1 메커니즘
거래일마다 ~278커밋이 영구 누적된다. 54일 만에 5,019커밋 = **월 ~5,800커밋(거래일 21일 × 278) 페이스**.

### 3.2 리스크
- `.git`은 현재 30MB로 작지만(커밋당 packed ~6KB), **선형 누적**이므로 연 환산 ~30~40MB/년 + 오브젝트 수 7만+/년 증가.
- 당장은 무해하나, 수년 뒤 clone/fetch 속도 저하, `git log`·blame 무거워짐, 오브젝트 수 팽창.
- `data/local/`은 gitignore라 이 팽창과 **무관**(별도 로컬 디스크 이슈, 누적1 범위 밖).

### 3.3 현재 상태
`git count-objects`: 6,280 오브젝트·30.79MB·**size-garbage 0**(정상). gc.auto 기본값. **몇 년은 여유.**

### 3.4 완화 선택지 (연 단위, 지금 아님)
1. **정기 `git gc --aggressive`** — pack 최적화(오브젝트 수는 유지되나 크기 압축).
2. **회차이력 커밋 자체를 git 밖으로** — 회차이력은 감사용이므로 `data/local/`(gitignore)이나 별도 저장소로 옮기면 커밋 팽창의 절반(~144/일)이 사라진다. 리스크 A·B를 동시에 완화하는 가장 효과적 조치.
3. 연 1회 히스토리 아카이브(오래된 데이터 커밋 squash) — 단, 협업/재현성 영향 있어 신중.
