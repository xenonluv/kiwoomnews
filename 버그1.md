## A. 확정 결함 5건 (즉시 수정 대상)

### 버그 1 — 미종결 매수 pending이 손절·강제청산까지 무기한 잠금 [최상]
- **위치**: `autotrade_monitor.py:447` `_blocked_by_pending_entry`
- **결함**: 같은 종목 pending이 비종결 상태(CANCEL_UNKNOWN·STATUS_INCONSISTENT·조회 반복 실패)로 남으면 만료·일자 전환·강제청산 예외 없이 해당 포지션의 `check_position` 자체를 continue — **실보유 주식의 손절과 익일 14:50 강제청산이 무기한 정지**, 알림 없음(로그만).
- **시나리오**: 1일차 15:18 매수 10주 중 4주 체결·잔량 취소가 타임아웃 → pending CANCEL_UNKNOWN 잔존 → 2일차 매분 틱이 청산 판정 이전에 스킵 → 실보유 4주가 폭락해도 침묵. (같은 파일 339행 `entry_price_verified` 게이트는 강제청산 **뒤에** 배치해 이 원칙을 지켰는데, 이 블록만 예외 조항이 없음 — 자체 불변조건과도 모순.)
- **수정 방향**: ① 강제청산(14:50) 분기는 pending 보류와 무관하게 항상 실행(339행과 동일한 배치) ② pending 보류에 시효 부여(예: 생성 후 N시간/일자 전환 시 에스컬레이션) ③ 보류 지속 시 텔레그램 재알림.

### 버그 2 — reconcile 예외 1건이 전 포지션 청산 판정을 통째로 중단 [최상]
- **위치**: `autotrade_monitor.py:424` `_run_unlocked`
- **결함**: `reconcile_pending_entries` 예외 시 틱 전체 return — 불량 pending 행 1개가 **무관한 전 포지션의 손절·강제청산을 매분 무기한 무력화**. 구코드(무조건 전 포지션 평가)에 없던 회귀. 알림 없음.
- **수정 방향**: reconcile 실패는 해당 pending만 격리(per-row try)하고 **check_position 루프는 반드시 진행**. 반복 실패 시 텔레그램 경고.

### 버그 3 — TP1 부분체결 시 본전방어·TP2 규칙 소실 [높음]
- **위치**: `autotrade_monitor.py:156`
- **결함**: 구코드는 TP1이 1주라도 체결되면 `tp1_done=True`(잔량에 본전방어 +0.5%·TP2 적용). 신코드는 전량 체결 시에만 설정 → 부분체결 잔량이 pre-TP1 분기에 남아 +0.6%로 밀려도 본전방어 없이 −5% 손절까지 보유.
- **수정 방향**: 체결 수량 ≥1이면 `tp1_done=True`(구코드 시맨틱 복원), 잔량은 기존 부분체결 처리 유지.

### 버그 4 — 제출 불명 경로의 저장이 무방비 (경고 유실 + 크래시) [높음]
- **위치**: `autotrade_executor.py:374` (동일 원인 :380)
- **결함**: SUBMIT_UNKNOWN 전이 후 `save_positions`가 예외를 던지면 텔레그램 경고 이전에 크래시 — 디스크엔 PREPARED/ord_no=None 잔존, 실제 접수된 주문이 체결되면 **원장 없는 실주식**(손절·강제청산·알림 전무).
- **수정 방향**: 경고를 저장보다 먼저 발송(또는 save+notify를 try로 감싸 알림 보장). 크래시 대신 다음 종목 진행.

### 버그 5 — SUBMIT_UNKNOWN 영구 래치가 재알림 없이 자동매매 전체를 조용히 정지 [높음]
- **위치**: `autotrade_orders.py:127`
- **결함**: ord_no 없는 SUBMIT_UNKNOWN/PREPARED 잔존 행은 어떤 자동 경로로도 해소 불가 + 최초 1회 알림뿐 → 회장님이 그 메시지 하나를 놓치면 **화·수·목… 신규 매수가 영구 차단**되는데 로그 외 아무 신호가 없음. (fail-closed 자체는 합의 정책 — 문제는 가시성.)
- **수정 방향**: ① 미해소 pending 존재 시 실행기 슬롯마다(또는 일 1회) 텔레그램 재알림 ② 수동 해소 절차 문서화(HTS 대조 후 pending 행 제거 명령) ③ 브로커 일마감으로 주문이 자연 소멸하는 날짜 경과 시 "만료 의심" 상태로 승격 표시.

### 버그 6 — KRX 매도(손절·TP·강제청산)의 kt00007 종결 확인 미실측 [실측 시 최상]
- `kiwoom_trade.py:132`. 신코드는 라이브 KRX 시장가 매도도 kt00007(dmst_stex_tp="KRX") 종결 확인으로만 원장 반영(구코드: 즉시 체결 처리). KRX측 조회가 행을 안 주면 **체결된 매도가 영구 pending** → 원장·계좌 괴리 + 다음날 전날포지션 가드가 전 매수 차단.
### 버그 7 — 제출 직후 0유예 전량취소 [실측 시 최상]
- `autotrade_executor.py:385` + `autotrade_orders.py:112-118`. 제출 수백 ms 후 같은 콜스택에서 reconcile → 체결 반영 지연이면 filled 0/remaining 10 → **갓 낸 주문을 즉시 전량 취소** — 그날 매수가 구조적으로 무효화. 수정3 §310은 "부분체결 확인 후 잔량 취소"를 요구했지 0초 취소가 아님. → 유예(최소 수 초~다음 틱) 또는 접수 직후 취소 금지 창 필요.
### 버그 8 — 0/0 응답을 CANCELLED로 판정해 중복매수 잠금 파기 [실측 시 최상]
- `autotrade_orders.py:124`. 등록 지연으로 cntr 0/remnq 0이 오면 취소로 오판·pending 삭제 → 원주문이 뒤늦게 체결되면 **원장 없는 실주식 + 같은 종목 재매수 허용**(이중 노출). → 0/0은 "미반영"으로 취급(N회 연속 확인 후에만 종결).
### 버그 9 — 평균체결가 필드명 3후보 추측 [실측 시 높음]
- `kiwoom_trade.py:166` + `autotrade_monitor.py:339`. 실필드가 제4의 이름이면 `entry_price_verified=False` 영구 → 전 포지션 자동 청산 보류(1회 알림). → 실측 체크리스트에 평균가 필드 포함 + 미검증 지속 시 시효 에스컬레이션.
### 버그 10 — 09시 전 거래일 판정이 센티널(005930) NXT 분봉에 의존 [실측 시 높음]
- `market_state.py:25`. NX 분봉에 당일 프리마켓 행이 안 찍히면 **08:00~09:00 프리마켓 손절 세션 전체가 매일 무동작**(이 기능의 존재 이유 상실). → 실측으로 08시대 NX 분봉 행 존재 확인, 부재 시 판정 소스 교체(예: NX 현재가 응답 시각).
