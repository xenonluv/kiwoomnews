## 2. 해석 정정 2건 (사실이되 결론이 다름)

### ② 문제 1 — "역전"은 통계적으로 유의하지 않다

B3 vs B4는 n=6 vs n=8. 윌슨 95% 구간이 각각 30~90%, 53~98%로 **거대하게 겹친다.** B6/B7/B8 역전도 동일. 즉 "버킷순이 틀렸다"는 확정은, 정렬4 합의에서 금지한 "소표본 명목치로 순서를 확정"하는 오류의 **역방향 재범**이다. 버킷이 사람이 정한 우선순위라는 점도 은폐된 결함이 아니라 정렬4.md에 명시된 설계다(회장님 40년 지문 프라이어 + 사전 고정 킬스위치 + 전진검증 후 조정). 실제로 실측 붕괴가 확인된 B10(급소 단독 잔여 20%)·B3 잔여 약세는 이미 강등 감시 대상으로 등재돼 있다.

**따라서 문제1.md 4단계 처방 중 "3단계: 정렬 점수 교체"는 기각한다.** 지금 바꾸면 이번 주 세 번째 재정렬이고, 7/10에 심은 전진검증 표본이 리셋된다. 정렬 조정은 `7월말봐라.md` 채점 절차(사전 고정 승격/강등 기준) 그대로 간다. 단 "검증된 확률을 정렬키로"라는 방향 자체는 7월말 채점에서 n≥20 버킷부터 단계 적용할 정당한 장기 목표로 인정한다.

### ② 문제 3 — 괴리의 원인은 '미갱신'이 아니라 '모집단 정의 불일치' (더 나쁨)

`RANK_BUCKET_BASELINES`의 기대치는 **코호트(겹침 포함) 통계**인데, 실제 버킷 배정은 **상호배타(가장 낮은 버킷 우선)**다. B10 카드의 66.7%는 급소 전체 코호트 수치이고, B10에 실제 배정되는 종목(상위 버킷 탈락 잔여)의 실측은 20%다. **이 괴리는 시간이 지나 생긴 드리프트가 아니라 도입 첫날부터 존재한 정의 오류**이며, 백테스트를 매일 돌려도 자동으로 안 고쳐진다. 씨이랩 재계산(기대 +17.39% → 실제 +5.79%)에서 보고한 "expected 낙관 편향"의 정량 확인. → 수정 가치 높음(§3의 P2).

## 3. 수용 항목과 우선순위 (codex 1~4단계 재배열)

정렬4 합의(전진검증 우선·수동 승인·자동매매 첫 주 동결)와 충돌하지 않는 순서로 재배열한다.

| 우선 | 항목 | 내용 | 규모 | 왜 이 순서인가 |
|---|---|---|---|---|
| **P1** | 문제 6 | ① suspect·history에 `rank_model_version`(예: "정렬4@7f3acab") 영구 저장 ② `rank_bucket_stats`를 **retro(소급 재분류)/forward(신호 시점 저장값)** 두 표로 분리 — forward는 저장된 rank_bucket만 사용 | 소 | **7월말 채점의 전제.** 지금 안 고치면 7월말 표도 소급·전진 혼합으로 오염되고, 규칙을 나중에 바꾸면 과거 forward 표본까지 소리 없이 재분류됨 |
| **P2** | 문제 3 | `RANK_BUCKET_BASELINES`의 expected를 **상호배타 소급 실측**으로 교체(B10 20%·B3 66.7% 등) + 갱신은 자동이 아니라 **backtest가 calibration 후보 파일 산출 → 회장님 승인 후 반영**(수동 승인 원칙 유지) | 소 | 카드에 첫날부터 틀린 기대치가 노출 중. 사용자 오도 방지 |
| **P3** | 문제 9 | history 쓰기를 임시파일→fsync→`os.replace` 원자 교체로 | 극소 | 10줄짜리 저비용 보험 |
| **P4** | 문제 7 | publish 재조립을 "원 배열 순서 유지 선택"으로 교체 + 전후 순서 동일성 회귀 테스트 | 극소 | 잠복 폭탄 제거. regular 경로 부활 시 즉사 방지 |
| **P5** | 문제 8 | `rank_bucket_stats`에 final/intraday(dropout 포함) 모집단 분리 열 추가 — 카드 expected는 final 기준 | 소 | 현재 표본에선 셀이 비지만 필드는 지금 넣어야 나중에 소급됨 |
| **P6** | 문제 2 | backtest에 신호일 단위 `rank_eval` 추가: top1_hit·top3_contains_winner·rank_spearman·rank별 평균 고가 | 소~중 | 유용하나 하루 suspects 1~3종목인 날이 많아 분별력 축적이 느림 — 기대치 관리 필요 |
| **P7** | 문제 4·5 | **범위 축소판**부터: 매 회차 전량 저장 대신 ① 일마감 1회 "전체 후보+게이트 통과/탈락 사유+컷 전 순위" 스냅샷(`data/radar_scan_history/YYYYMMDD.json`) ② 게시 suspects에 한해 회차별 rank 경로 append(`rank_path`) ③ 폭발 레지스트리 활성 종목의 당일 분봉 스파크 요약 보존(분봉은 소급 불가 — 데이타솔루션 사례) | 중~대 | 선택편향·false negative 검증의 근본 해결이나 저장량·복잡도 큼. 축소판으로 시작해 효용 확인 후 확장 |

**기각**: 3단계 "버킷 정렬 제거·확률 정렬키 즉시 교체"(§2-① — 7월말 채점 후 판단), "성과 악화 시 kill switch가 자동으로 정렬 변경"(자동튜닝 금지 원칙 — 킬스위치는 현행대로 **자동 판정·수동 상신**).

## 4. 실행 조건 (의뢰 수령 시)

- 구현 순서 P1→P7, 각 단계 py_compile + 기존 스모크/회귀 테스트 확장 + 적대 코드리뷰(P1~P4는 1회, P7은 2회) + publish dry-run + 라이브 1회차 확인 후 커밋.
- 정렬 순서(rank_bucket 체계)는 **한 글자도 바꾸지 않는다** — 이번 의뢰는 저장·계측·표시 정확성 계층만.
- 자동매매 관련 코드 무접촉. cron이 워킹트리를 실행하므로 P3(원자 쓰기)부터 적용해 이후 작업의 안전판으로 삼는다.
- 완료 기준: forward 표(신호 시점 버킷)가 7/10 이후 표본만으로 집계되고, 카드 expected가 상호배타 기준으로 바뀌었으며, 스캔 스냅샷 1일치가 실제로 쌓이는 것을 라이브로 확인.
