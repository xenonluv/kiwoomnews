# 크리티컬 이슈 1~5 수정 계획

작성 기준: 2026-07-11 현재 `/Users/jinjin/kiwoomnews` 운영 코드

이 문서는 앞선 코드 리뷰에서 확인한 다섯 가지 문제를 실제로 고치는 순서와 검증 방법을 정리한다. 이 문서 작성만으로 운영 코드는 바뀌지 않는다. 자동매매의 주문·수량·순위 정책은 검증 없이 함께 변경하지 않는다.

## 0. 전체 원칙과 실제 적용 순서

### 반드시 지킬 불변조건

1. 주문 API의 성공 응답은 `체결`이 아니라 `접수 가능성`으로 취급한다.
2. 브로커 주문보다 먼저 중복 주문을 막는 상태를 디스크에 영구 저장한다.
3. 동일 계좌의 매수 실행기와 청산 모니터는 동시에 포지션 원장을 수정할 수 없다.
4. 주문 결과가 불명확하면 자동 재주문하지 않고 잠금을 유지하며 수동 확인을 요청한다.
5. 연구 데이터 저장 실패는 기존 레이더 순위와 주문 판단을 바꾸지 않는다.
6. 탈락 종목의 연구 결과를 곧바로 운영 순위에 반영하지 않는다.
7. 실제 거래일 여부가 불명확하면 게시와 주문 모두 fail-closed 한다.

### 권장 적용 순서

이슈 번호 순서와 실제 배포 순서는 다르게 잡는다. 안전장치를 먼저 설치한 뒤 주문 체결 로직을 바꾸는 편이 안전하다.

1. **3번**: cron 설치기의 기본 LIVE 문제 제거
2. **2번**: 자동매매 프로세스 전역 잠금 도입
3. **1번**: 매수 주문의 접수·체결 분리
4. **5번**: 휴장일과 데이터 기준일 검증
5. **4번**: 비게시 관찰 종목의 익일 결과 연결

각 단계는 독립 커밋으로 만들고, 해당 단계의 테스트와 DRY 운영 확인이 끝나기 전 다음 단계로 넘어가지 않는다.

### 작업 시작 전

1. 웹 자동매매 토글을 OFF로 둔다.
2. 현재 crontab과 자동매매 원장을 백업한다.

   ```bash
   crontab -l > /tmp/kiwoomnews-cron-before-fix.txt
   cp -p data/autotrade_positions.json /tmp/autotrade_positions.before-fix.json 2>/dev/null || true
   cp -p data/autotrade_trades.jsonl /tmp/autotrade_trades.before-fix.jsonl 2>/dev/null || true
   ```

3. 실계좌 보유수량, 미체결 주문, `data/autotrade_positions.json`을 사람이 한 번 대조한다.
4. 수정 중 cron이 워킹트리를 직접 실행하지 않도록 별도 브랜치에서 작업하거나, 최소한 자동매매 cron을 DRY로 다시 설치한다.
5. 기존 스모크 테스트를 먼저 실행해 기준선을 남긴다.


## 1. 매수 주문 접수와 실제 체결을 분리한다

### 현재 문제

`scripts/kiwoom_trade.py`의 매수 함수는 주문 접수 응답 뒤 요청 수량과 참고가격을 반환한다. `scripts/autotrade_executor.py::_record()`는 이를 전량 체결로 간주해 즉시 `status=open`, `qty_open=요청수량`으로 저장한다.

특히 NXT 지정가 매수는 미체결 또는 부분체결 가능성이 높다. 10주 주문 중 4주만 체결돼도 현재 원장은 10주를 보유한 것으로 기록한다. 이후 예산 계산, 중복매수 방지, 손절·익절 수량이 모두 틀어진다.

### 목표 상태

주문 상태를 다음처럼 명시적으로 구분한다.

```text
PREPARED -> SUBMIT_UNKNOWN | ACCEPTED -> PARTIAL | FILLED | CANCELLED | REJECTED
```

- `PREPARED`: 브로커 호출 전에 디스크에 저장된 주문 의도
- `SUBMIT_UNKNOWN`: 타임아웃 등으로 주문 접수 여부를 알 수 없음
- `ACCEPTED`: 주문번호를 확보했으나 체결은 미확정
- `PARTIAL`: 일부 수량만 체결
- `FILLED`: 체결수량 전부 확인
- `CANCELLED/REJECTED`: 미체결 잔량이 종결됨

`positions[]`에는 확인된 체결수량만 들어간다. 미확정 주문은 별도 `pending_entries[]`에 둔다.

### 단계별 수정

#### 1-1. 주문 조회 필드를 먼저 실측한다

1. 키움 공식 문서와 DRY가 아닌 소액 주문 응답에서 다음 필드명을 확인한다.
   - 주문번호
   - 매수/매도 구분
   - 주문수량
   - 누적 체결수량
   - 미체결수량
   - 체결가격 또는 체결금액
   - 거부·취소 상태
2. 현재 `order_status()`는 매도 조회 조건에 맞춰져 있으므로 매수 조회에 그대로 쓰지 않는다.
3. 확인된 원본 응답에서 시크릿을 제거한 fixture를 테스트 파일에 저장한다.
4. 필드가 불확실하면 구현을 추정하지 말고 해당 단계에서 중단한다.

#### 1-2. 주문 상태 조회를 매수·매도 공용으로 일반화한다

1. `kiwoom_trade.order_status()`에 `side="buy"|"sell"`을 명시하게 한다.
2. 반환값을 다음 공통 구조로 정규화한다.

   ```python
   {
       "ord_no": "...",
       "side": "buy",
       "ordered_qty": 10,
       "filled_qty": 4,
       "remaining_qty": 6,
       "filled_amount": 50000,       # API가 제공할 때
       "avg_fill_price": 12500,      # 원본 체결정보로 계산 가능할 때
       "terminal": False,
       "rejected": False,
       "cancel_confirmed": False,
   }
   ```

3. 필수 수량 필드가 없거나 `filled + remaining > ordered`이면 예외로 처리해 잠금을 유지한다.
4. 원주문번호가 같은 다른 주문과 섞이지 않도록 종목코드·매매구분·주문일도 함께 대조한다.

#### 1-3. 포지션 원장에 `pending_entries`를 추가한다

원장 루트 구조를 하위호환 방식으로 확장한다.

```json
{
  "schema_version": 2,
  "positions": [],
  "pending_entries": []
}
```

각 pending에는 최소한 다음을 저장한다.

- 고유 intent ID
- 종목코드와 이름
- KRX/NXT, 슬롯, 매수 방향
- 요청수량과 지정가/참고가
- 배정예산
- 주문 전 결정 스냅샷 메타데이터
- `state`, `ord_no`, `ordered_qty`, `accounted_filled`
- 생성·최근조회 시각과 주문일
- 오류 메시지와 수동 확인 필요 여부

구버전 원장을 읽을 때 `pending_entries=[]`를 자동 기본값으로 보완하되, 파일 내용을 조용히 버리지 않는다.

#### 1-4. 주문 전에 pending을 영구 저장한다

1. 실행기는 종목별 주문 직전에 `PREPARED` pending을 저장한다.
2. 저장에 실패하면 브로커 주문을 호출하지 않는다.
3. 그 다음 주문을 한 번만 호출한다. 주문 API 자체는 자동 재시도하지 않는다.
4. 응답에서 주문번호를 얻으면 `ACCEPTED`로 즉시 다시 저장한다.
5. 네트워크 타임아웃처럼 접수 여부가 불명확하면 `SUBMIT_UNKNOWN`으로 저장하고 자동 재주문을 금지한다.
6. 주문번호를 얻지 못한 상태는 계좌·주문내역 수동 확인 전까지 잠금을 풀지 않는다.

#### 1-5. 확인된 체결수량만 포지션으로 전환한다

1. 별도 `reconcile_pending_entries()`가 주문번호로 상태를 조회한다.
2. `newly_filled = filled_qty - accounted_filled`만 새 체결로 반영한다.
3. 부분체결이면 확인된 수량만 포지션에 더하고 pending은 유지한다.
4. 잔량 취소 후 `remaining_qty=0`이 확인돼야 pending을 종결한다.
5. 실제 진입가는 주문 참고가격이 아니라 체결가중평균을 사용한다.
6. 체결가격을 API로 신뢰성 있게 얻지 못하면 `entry_price_verified=false`로 표시하고 자동 손익 청산을 시작하지 않는다. 계좌 평균단가 조회로 교차확인한 뒤 활성화한다.
7. 같은 코드에 기존 수동 보유분이 있어도 봇 체결분만 별도 수량으로 기록한다.

#### 1-6. KRX와 NXT를 동일 원칙으로 처리한다

- KRX 시장가도 접수 즉시 체결됐다고 가정하지 않는다.
- NXT 지정가는 부분체결과 잔량 취소를 정상 시나리오로 취급한다.
- 다음 모니터 회차가 pending을 먼저 재조정한 뒤 손절·익절을 판단하게 한다.
- 미확정 매수 pending이 있으면 같은 종목의 신규 매수를 차단한다.

### 필수 테스트

1. NXT 10주 주문, 0주 체결 후 전량 취소
2. NXT 10주 주문, 4주 부분체결 후 6주 취소
3. NXT 10주 전량 체결
4. KRX 시장가 주문 접수 후 조회 지연
5. 주문 제출 타임아웃 후 실제로는 접수된 경우
6. 주문번호 추출 실패
7. pending 저장 실패 시 브로커 함수 미호출
8. 프로세스 재시작 후 주문번호로 복구
9. 수동 보유 50주와 봇 신규 체결 4주의 분리
10. 실제 체결수량만 `deployed_today`, `already_bought`, 청산수량에 반영되는지 확인

### 완료 조건

- 코드 어디에도 주문 응답의 요청수량을 곧바로 `qty_open`으로 기록하는 경로가 없어야 한다.
- 부분체결 fixture로 재시작 테스트가 통과해야 한다.
- 소액 DRY/모의 검증 후 실계좌 1주 주문에서 주문내역·계좌·원장 수량이 일치해야 한다.

## 2. 자동매매 포지션 원장에 프로세스 전역 잠금을 적용한다

### 현재 문제

`load_positions()`와 `save_positions()`는 각각 파일 읽기와 원자 교체만 수행한다. 전체 `읽기 -> 판단 -> 주문 -> 저장` 트랜잭션에는 프로세스 잠금이 없다. 1분 cron 모니터가 겹치거나 15:18 실행기와 모니터가 동시에 시작하면 같은 포지션에 중복 주문을 낼 수 있다.

### 목표 상태

동일 계좌에 주문할 수 있는 프로세스는 한 시점에 하나뿐이어야 한다. 잠금을 얻지 못한 cron 회차는 기다리며 겹치기보다 즉시 종료하고 다음 회차에 맡긴다.

### 단계별 수정

#### 2-1. 단일 자동매매 실행 잠금을 정의한다

1. `data/.autotrade.execution.lock` 또는 `/tmp/kiwoomnews-autotrade.lock` 하나를 SSOT로 정한다.
2. `autotrade_common.py`에 `acquire_execution_lock(nonblocking=True)` context manager를 만든다.
3. Mac/Linux에서는 `fcntl.flock(LOCK_EX | LOCK_NB)`를 사용한다.
4. Windows를 계속 지원한다면 `msvcrt.locking` 또는 Task Scheduler의 중복 실행 금지 설정을 함께 사용한다. 단, 운영 안전을 OS 설정에만 의존하지 않는다.
5. 잠금 파일에는 현재 PID, 시작시각, 실행 종류를 진단용으로 기록하되 flock 자체를 권위로 삼는다.

#### 2-2. 실행기와 모니터가 같은 잠금을 사용하게 한다

1. `autotrade_executor.run()` 진입 직후 잠금을 잡는다.
2. `autotrade_monitor.run()`도 같은 잠금을 잡는다.
3. 잠금을 얻지 못하면 `다른 자동매매 프로세스 실행 중 — 이번 회차 skip`을 로그에 남기고 주문 없이 종료한다.
4. 계좌조회·주문·pending 저장·포지션 저장이 끝날 때까지 잠금을 유지한다.
5. 예외가 발생해도 context manager가 잠금을 반드시 해제하게 한다.

#### 2-3. 포지션 파일 쓰기를 강화한다

1. 임시파일명을 고정 `.tmp`가 아니라 `.tmp.<pid>.<uuid>`로 만든다.
2. `json.dump -> flush -> fsync -> json 재파싱 -> os.replace -> 디렉터리 fsync` 순서로 저장한다.
3. 저장 전 원장 스키마와 수량 불변조건을 검사한다.
   - `qty_open >= 0`
   - closed 포지션은 `qty_open == 0`
   - 같은 주문번호가 pending에 중복되지 않음
   - `accounted_filled <= ordered_qty`
4. 저장 실패 시 기존 파일이 그대로 유지되는지 테스트한다.

#### 2-4. 잠금 범위를 우회하는 함수 호출을 없앤다

1. 운영 진입점 외부에서 `_sell()`이나 `_record()`를 직접 호출하지 않게 내부 함수로 취급한다.
2. 모든 주문 함수가 현재 실행 잠금 보유 여부를 검증하도록 선택적 assertion을 둔다.
3. `autotrade_reconcile.py`처럼 포지션을 바꾸는 다른 스크립트도 동일 잠금을 사용한다.
4. 통계처럼 읽기만 하는 작업은 스냅샷을 짧게 읽고 주문 잠금을 오래 점유하지 않는다.

### 필수 테스트

1. 두 모니터 프로세스를 동시에 시작해 매도 주문이 정확히 한 번만 호출되는지 확인
2. 모니터 실행 중 매수 실행기를 시작하면 하나가 안전하게 skip 되는지 확인
3. 첫 프로세스가 예외 종료된 뒤 OS가 잠금을 회수하고 다음 회차가 실행되는지 확인
4. 동시에 서로 다른 종목을 처리해도 원장 갱신이 유실되지 않는지 확인
5. 저장 도중 강제 예외를 내 기존 원장이 손상되지 않는지 확인
6. 1분보다 오래 걸리는 가짜 API를 넣어 다음 cron 회차가 주문 없이 종료되는지 확인

### 완료 조건

- 실행기·모니터·reconcile 어디에서도 공용 잠금 없이 주문을 호출할 수 없어야 한다.
- 실제 crontab을 유지한 채 인위적으로 70초 지연시킨 테스트에서 중복 주문이 0건이어야 한다.

## 3. cron 설치기의 기본 LIVE 활성화를 제거한다

### 현재 문제

`install_cron_kiwoom.sh`는 기본 DRY이고 `--live`가 필요하지만, `install_cron.sh`는 옵션 없이도 `AUTOTRADE_LIVE=1`을 넣는다. 여러 운영 문서는 후자를 재실행하라고 안내한다.

### 목표 상태

설치 명령의 기본값은 항상 DRY여야 하고, LIVE는 사용자가 그 실행에서 명시적으로 요청해야 한다. 설치기가 두 개라면 동작이 완전히 같아야 한다.

### 단계별 수정

#### 3-1. 설치기 SSOT를 하나로 통합한다

1. `install_cron_kiwoom.sh`를 권위 설치기로 정한다.
2. `install_cron.sh`는 직접 cron을 만들지 말고 권위 설치기로 인자를 전달하는 얇은 wrapper로 바꾼다.
3. 두 스크립트 모두 다음 인터페이스만 허용한다.
   - 인자 없음: DRY 설치
   - `--live`: LIVE 설치
   - `--dry-run`: DRY 블록 미리보기
   - `--live --dry-run`: LIVE 블록 미리보기
   - `--uninstall`: 프로젝트 블록 제거
4. 알 수 없는 옵션은 비정상 종료한다.

#### 3-2. 현재 모드를 설치 결과에 명확히 표시한다

1. 설치 전후에 `자동매매 실발주: ON/OFF`를 굵은 경고 문구로 출력한다.
2. LIVE 설치 시 `AUTOTRADE_LIVE=1`이 들어가는 정확한 세 줄을 보여준다.
3. 기존 LIVE 블록을 인자 없는 명령으로 재설치하면 DRY로 내려가는 것이 안전 기본임을 문서화한다.
4. KV 토글은 두 번째 게이트일 뿐 설치기의 안전 기본을 대신하지 않는다고 명시한다.

#### 3-3. 모든 문서를 같은 명령으로 고친다

다음 문서의 설치 안내를 검색해 모두 통일한다.

- `README.md`
- `CLAUDE.md`
- `HANDOFF.md`
- `업데이트mac.md`
- `7월말봐라.md`
- `맥설치매뉴얼.md`

운영 절차는 `기본 DRY 설치 -> 스모크 테스트 -> 계좌 대조 -> --live 설치 -> KV 토글 ON` 순서로만 안내한다.

### 필수 테스트

1. 두 설치기를 인자 없이 `--dry-run` 상당으로 실행했을 때 `AUTOTRADE_LIVE=1`이 없어야 한다.
2. `--live --dry-run`에서만 `AUTOTRADE_LIVE=1`이 나타나야 한다.
3. 설치기를 두 번 실행해도 프로젝트 cron 블록이 하나만 존재해야 한다.
4. 다른 저장소 cron 라인이 삭제되지 않아야 한다.
5. `--uninstall`이 키움 블록만 제거해야 한다.

### 완료 조건

```bash
rg -n 'install_cron\.sh|install_cron_kiwoom\.sh' README.md CLAUDE.md HANDOFF.md *.md
```

검색된 모든 실제 설치 안내가 동일한 안전 기본을 설명해야 한다. 아무 인자 없이 설치했을 때 실발주 환경변수가 cron에 들어가면 실패다.

## 4. 비게시 관찰 종목에도 익일 결과를 연결한다

### 현재 문제

`data/local/radar_raw/.../scans`에는 `REJECT_RULE`, `OUT_OF_UNIVERSE`, `MISSING_DATA`까지 저장되지만 `radar_backtest.evaluate()`는 `data/radar_history`의 게시 후보만 평가한다. 따라서 데이타솔루션·씨이랩 같은 비게시 종목은 저장만 되고 회전율별 익일 고가 연구 표본으로 자동 연결되지 않는다.

### 목표 상태

게시 성과와 연구 모집단을 분리한 채, 하루의 대표 관찰 스냅샷에 있는 종목을 익일 OHLC와 연결한다. 이 연구 결과는 운영 순위·가중치·자동매매에 자동 반영하지 않는다.

### 단계별 수정

#### 4-1. 하루에 어떤 스캔을 평가할지 먼저 고정한다

매 회차 239개를 모두 평가하면 같은 종목이 중복된다. 다음 중 하나를 명시적으로 SSOT로 정한다.

- 권장: KRX 15:30 이전 마지막 성공 스캔을 `research_krx_close` 모집단으로 고정
- 별도: NXT 19:50 이전 마지막 스캔은 `research_nxt`로 분리
- 별도: 운영 EOD 마지막 스캔은 데이터 품질 감사용으로만 사용

`scan_ok=true`, `dry_run=false`, 데이터 기준일이 신호일과 같은 스캔만 선택한다. 선택한 run ID와 생성시각을 불변 cohort 파일에 저장한다.

#### 4-2. 연구용 일별 cohort를 만든다

새 파일을 기존 게시 history와 분리한다.

```text
data/local/radar_raw/YYYY/MM/DD/research/krx_close_cohort.json
```

종목코드별로 다음을 보존한다.

- 원본 run ID와 관찰시각
- 상태와 모든 gate 탈락 사유
- KRX 종가 기준 가격 스냅샷
- 회전율, 거래대금, 고가상승률, fade
- 전일 폭발 이력
- 스파크 요약
- 분봉 파일 참조와 fetch 상태
- `eligible_for_forward_eval` 및 제외 사유

`API_ERROR`, 기준일 불일치, 종가 없음은 결과 표본에서 제외하되 제외 건수와 이유는 유지한다. `REJECT_RULE`과 정상적인 `OUT_OF_UNIVERSE`는 연구 표본에 포함한다.

#### 4-3. 익일 결과를 별도 파일에 기록한다

기존 게시 후보용 `evaluation/next_day.json`을 덮어쓰지 않는다.

```text
evaluation/observed_union_next_day.json
```

각 종목에 다음을 기록한다.

- 신호일 확정 종가와 기준
- 실제 다음 거래일
- 익일 시가·고가·저가·종가
- 시가·고가·저가·종가 수익률
- +3/+7/+11/+13/+15% 터치
- -3/-5/-7/-10% 터치
- 일봉만 있을 때 고가·저가 발생 순서는 `unknown`
- 평가 성공·보류·만료 상태와 이유

신호일 봉이 없거나 다음 거래일 봉이 아직 없으면 평가하지 않고 재시도한다. 휴일을 단순히 달력 `+1일`로 계산하지 않는다.

#### 4-4. API 부하와 중복을 제한한다

1. 같은 종목·같은 실행의 일봉 조회는 캐시한다.
2. cohort당 종목코드를 dedup한다.
3. API 오류는 전체 평가 실패로 만들지 말고 해당 종목을 보류한다.
4. 한 실행의 최대 호출 수와 처리시간을 로그에 남긴다.
5. 원본 scan과 cohort checksum이 바뀌지 않았는지 평가 전 검증한다.

#### 4-5. 연구 통계를 운영 통계와 분리한다

새 출력 예:

```text
web/data/research_turnover_performance.json
```

이 파일은 우선 웹에 노출하지 않아도 된다. 다음 구간별로 `n`, 고유종목 수, 익일 고가 중앙값, 중간 50% 범위, +7/+11/+13/+15%, -5%, +7/-5 동시 터치를 계산한다.

- 0~5%
- 5~10%
- 10~20%
- 20~40%
- 40~80%
- 80~150%
- 150~300%
- 300% 이상

당일 등락률·fade·장 후반 스파크 등 조건을 추가로 나눌 때는 작은 표본을 숨기거나 `low confidence`로 표시한다. 연구 결과를 `radar_weights.json`이나 `rank_policy.py`에 자동 적용하지 않는다.

### 필수 테스트

1. 데이타솔루션 유형 `REJECT_RULE`이 cohort와 익일 결과에 포함되는지 확인
2. 씨이랩 유형이 게시 history에 없어도 평가되는지 확인
3. 같은 종목이 여러 회차에 있어도 하루 한 표본만 생성되는지 확인
4. `API_ERROR`, `MISSING_DATA`가 성공 표본으로 섞이지 않는지 확인
5. 게시 후보의 기존 성과 숫자가 전후 동일한지 확인
6. 연구 파일 저장 실패가 레이더·자동매매 결과를 바꾸지 않는지 확인
7. 휴일 뒤 실제 다음 거래일이 정확히 연결되는지 확인

### 완료 조건

- 7월 10일 fixture에서 데이타솔루션과 씨이랩이 `observed_union_next_day` 평가 대상으로 나타나야 한다.
- 기존 `performance.json`과 운영 rank 결과는 이 기능 추가 전후 동일해야 한다.
- 최소 20거래일 축적 전에는 연구 결과로 운영 문턱을 자동 변경하지 않는다.

## 5. 휴장일과 데이터 기준일을 검증한다

### 현재 문제

`kiwoom_client.minute_bars_today()`는 벽시계의 오늘이 아니라 응답에서 가장 최신 거래일을 선택한다. cron과 `publish.market_session()`은 평일 여부만 보므로 평일 휴장일에는 전 거래일 분봉을 오늘 데이터처럼 재사용할 수 있다.

### 목표 상태

라이브 레이더에서는 모든 핵심 가격·분봉 데이터의 거래일이 KST 오늘과 같아야 한다. 거래일을 확인하지 못하거나 시장이 휴장이라면 기존 정상 레이더를 stale 표시로 보존하고 새 신호·주문·연구 표본을 만들지 않는다.

### 단계별 수정

#### 5-1. API 반환값에 데이터 기준일을 노출한다

1. `minute_bars_today()`가 내부에서 알아낸 `latest_day`를 버리지 않게 한다.
2. 기존 호출 호환성을 위해 다음 중 하나를 선택한다.
   - `minute_bars_today_with_meta()`를 새로 만들어 `{trade_date, bars, fetch_status}` 반환
   - 기존 함수에 `with_meta=False` 옵션 추가
3. 라이브 radar 경로는 반드시 meta 반환 함수를 사용한다.
4. 일봉·현재가도 가능한 경우 `data_trade_date`를 함께 반환하게 한다.

#### 5-2. 중앙 거래일 검증 함수를 만든다

`market_state.py` 같은 단일 모듈에서 다음을 판정한다.

```python
{
    "kst_date": "20260717",
    "is_weekday": True,
    "broker_trade_date": "20260716",
    "is_trading_day": False,
    "market_open_confirmed": False,
    "reason": "broker_latest_trade_date_mismatch"
}
```

요일만으로 거래일을 확정하지 않는다. 우선순위는 다음과 같이 둔다.

1. 거래소 또는 브로커의 시장상태/영업일 응답
2. 당일 유효 일봉·분봉 존재와 응답 거래일 일치
3. 로컬 휴장 캘린더는 보조 안전망

서로 모순되면 열림으로 추정하지 말고 fail-closed 한다.

#### 5-3. 레이더에서 기준일 불일치를 장애로 처리한다

1. `radar.py` 시작 시 예상 거래일을 KST 오늘로 고정한다.
2. 분봉의 `trade_date != expected_trade_date`이면 빈 정상 데이터가 아니라 `STALE_TRADE_DATE` 오류로 기록한다.
3. 가격·일봉·분봉 중 핵심 기준일이 다르면 `scan_ok=false`로 종료한다.
4. `publish.py`는 이 경우 빈 레이더를 새로 게시하지 않는다.
5. 기존 `web/data/radar.json`은 유지하되 `stale` 경고를 운영 로그와 알림에 남긴다.
6. 휴장일 scan은 연구 cohort로 만들지 않는다.

#### 5-4. 자동매매가 독립적으로 거래일을 재검증하게 한다

레이더 검증만 믿지 않는다.

1. 매수 실행기 주문 직전에 `is_trading_day`와 해당 세션 열림을 확인한다.
2. 청산 모니터도 세션 판정 전에 동일 상태를 확인한다.
3. 불확실하면 주문 없이 종료한다.
4. `AUTOTRADE_FORCE_*` 테스트 훅은 테스트 환경에서만 작동하게 하고 운영 로그에 강제모드를 명확히 남긴다.
5. 휴장일에 API가 전일 가격을 반환해도 손절·익절 계산을 실행하지 않는다.

#### 5-5. historical 조회와 live 조회를 분리한다

- 연구·백필 함수는 명시적 `trade_date`를 받아 과거 최신일을 조회할 수 있다.
- 라이브 함수는 `allow_previous_day=False`가 기본이어야 한다.
- `today`라는 이름의 함수가 전일 데이터를 반환하지 않게 계약을 명확히 한다.

### 필수 테스트

1. 평일 휴장일에 키움이 전일 분봉을 반환하는 fixture
2. 토요일·일요일
3. 정상 평일 09:01의 당일 분봉
4. 정상 평일 장전으로 당일 봉이 아직 없는 경우
5. 일봉은 오늘인데 분봉은 전일인 혼합 응답
6. 휴장일에 publish가 기존 radar를 빈 파일로 덮어쓰지 않는지 확인
7. 휴장일에 매수·매도 함수가 한 번도 호출되지 않는지 확인
8. 연휴 뒤 첫 거래일에 정상 복귀하는지 확인

### 완료 조건

- weekday만으로 `market_session=open`을 확정하는 운영 경로가 없어야 한다.
- 라이브 radar의 저장 원본에 `expected_trade_date`, `actual_trade_date`, `market_state`가 남아야 한다.
- 평일 휴장 fixture에서 게시·매수·매도·forward cohort 생성이 모두 0건이어야 한다.

2. `--live` 설치 전 crontab 미리보기를 확인한다.
3. 매수 주문번호·실체결수량·평균단가·pending 종결·포지션 원장을 HTS와 대조한다.
4. 부분체결을 강제로 만들 수 없다면 낮은 체결 가능 지정가로 접수 후 취소 흐름을 검증한다.
5. 주문결과 불명 시 자동 재주문이 없는지 확인한다.
6. 매도 역시 주문번호와 실제 체결수량이 일치하는지 확인한다.

### 6-4. 롤백 기준

다음 중 하나라도 발생하면 KV 토글 OFF 후 즉시 DRY cron으로 되돌린다.

- 원장수량과 계좌수량 불일치
- 동일 intent 또는 같은 포지션의 중복 주문
- 주문번호 없는 pending이 자동으로 해제됨
- 부분체결 수량보다 많은 포지션 기록
- 휴장일 또는 기준일 불일치 데이터로 신호 생성
- 비게시 연구 통계가 운영 순위나 가중치를 변경
- manifest checksum 오류 또는 원장 JSON 손상

롤백은 코드만 되돌리는 것으로 끝내지 않는다. 미체결 주문과 실계좌 잔고를 먼저 수동 확인하고, pending 및 position 원장을 실제 계좌에 맞춰 복구한 뒤 자동매매를 재개한다.

## 7. 최종 완료 체크리스트

- [ ] 설치기 기본값이 모두 DRY다.
- [ ] 실행기와 모니터가 동일한 전역 잠금을 쓴다.
- [ ] 매수 주문 전에 pending intent가 영구 저장된다.
- [ ] 주문 접수와 체결이 분리돼 있다.
- [ ] 확인된 체결수량과 체결가만 포지션에 반영된다.
- [ ] 제출 결과 불명 주문은 자동 재주문되지 않는다.
- [ ] 휴장일과 데이터 기준일 불일치가 fail-closed 된다.
- [ ] 비게시 관찰 종목의 대표 일별 cohort가 중복 없이 생성된다.
- [ ] 비게시 익일 결과가 별도 연구 파일에 저장된다.
- [ ] 연구 결과가 운영 순위·자동매매에 자동 반영되지 않는다.
- [ ] 기존 테스트와 신규 적대 테스트가 모두 통과한다.
- [ ] 최소 1거래일 DRY 관찰과 1주 소액 LIVE 대조가 끝났다.

이 체크리스트가 모두 충족되기 전에는 자동매매 안전성과 비게시 종목 학습 체계가 완성됐다고 판단하지 않는다.
