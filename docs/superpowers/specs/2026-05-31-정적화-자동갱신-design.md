# 무료·고동시성 자동갱신 (정적화 + 엣지캐시 + 클라이언트 폴링) — 설계

- 날짜: 2026-05-31
- 목표: 깜빡임 없는 자동 갱신 + 1000명+ 동시접속에도 Vercel **무료(Hobby)** 한도 안에서 안정.
- 비목표: 데이터 파이프라인(cron/publish.py) 변경 없음. 게시 주기(15분)·데이터 형식 그대로.

## 배경 / 핵심 관찰

`web/lib/signals/repository.ts`가 `signals.json`을 **빌드 시점에 import**한다 → 데이터는 **배포할 때만**(=cron push마다) 바뀐다. 따라서 페이지를 `force-dynamic`으로 둘 이유가 없다. 정적(CDN)으로 바꾸면 방문자는 서버 함수를 거치지 않으므로 동시접속이 늘어도 함수 호출·비용이 거의 0이다.

현재 `force-dynamic`을 쓴 이유는 홈/상세가 `signalService`(HTTP self-fetch + `getBaseUrl()`의 `next/headers`)를 거치기 때문 → `headers()`가 동적 렌더를 강제. 정적화하려면 서버 컴포넌트에서 **repository를 직접 읽어** 렌더한다.

## 아키텍처

```
[방문자 N명] ──HTML·데이터── Vercel CDN(엣지 정적 캐시) ──(함수 ~0회)
                                  ▲
                                  │ 배포마다 1회 갱신
                          cron push → 재빌드
[열린 탭] ─60초 폴링→ /api/signals (엣지캐시 s-maxage) → 변경시만 상태 교체(무깜빡임)
```

## 변경 사항

### 1. API 라우트 엣지 캐시 (`app/api/signals/route.ts`, `app/api/signals/[post_id]/route.ts`)
- `export const dynamic = "force-dynamic"` 제거.
- 성공 응답에 헤더: `Cache-Control: public, s-maxage=30, stale-while-revalidate=300`.
- 효과: 1000명이 동시 폴링해도 CDN이 캐시 응답 → 실제 함수는 ~30초당 1회. 새 배포 시 Vercel이 캐시 자동 무효화 → 다음 폴링에 새 데이터.

### 2. 홈 정적화 + 라이브 폴링 (`app/page.tsx`, 신규 `components/signal/LiveSignals.tsx`, 신규 `services/signal.client.ts`)
- `page.tsx`: `force-dynamic` 제거. `signalService`(HTTP) 대신 **`listSignals()` repository 직접 호출**(빌드 시 정적). 초기 데이터를 `<LiveSignals initial=... />`에 전달.
- `LiveSignals.tsx` (`"use client"`): 초기 데이터로 두 섹션(📌시그널/👀후보)+`SignalCard`(Link 래핑) 렌더. **60초마다** client service로 `/api/signals?limit=50` 재요청, JSON이 실제로 바뀐 경우에만 `setState` → React 키(post_id) 재조정으로 부드럽게 교체(전체 리로드 없음 = 무깜빡임).
- `services/signal.client.ts`: 브라우저용 fetch(상대경로 `/api/signals`, `next/headers` 미사용). "컴포넌트 직접 fetch 금지" 규칙을 지키도록 서비스 계층 경유.

### 3. 상세 정적화 (`app/signals/[post_id]/page.tsx`, `lib/signals/repository.ts`)
- `force-dynamic` 제거. `getSignal()` 직접 호출.
- `generateStaticParams()`로 현재 배포의 모든 `post_id` 사전 생성 → 전부 정적.
- `export const dynamicParams = false` → 미생성 id는 정적 404(함수 0회). 모든 유효 id는 배포에 포함되므로 안전.
- repository에 `allSignalIds()` 추가.

## 무료 한도 영향

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 페이지 방문 | 방문마다 SSR 함수 1회 | CDN 정적 0회 |
| 상세 방문 | 방문마다 SSR | 정적(사전생성) 0회 |
| 폴링(탭당 60초) | — | 엣지캐시 → 전역 ~30초당 함수 1회 |

→ 동시접속 1000명+ 이어도 함수/대역폭이 무료 한도에 거의 영향 없음.

## 검증

1. `cd web && npm run build` → 라우트 표에 `/`, `/signals/[post_id]`가 **○(Static)** 또는 SSG로 표기되는지 확인.
2. 로컬 prod 서버: 홈 정상 렌더 + 카드/뉴스 유지, 상세 정상.
3. 폴링: signals.json을 임시 변경 후 `/api/signals` 응답이 바뀌는지(헤더 `Cache-Control` 포함) 확인.
4. 커밋 → push → 라이브에서 홈/상세 200, 응답 헤더 확인.
