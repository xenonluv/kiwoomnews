### 심각도 보정 (이슈 5)

기존 가드가 최악 시나리오를 상당 부분 차단 중:
- `radar._allow_signal_date`: 평일 09시 이후 `signal_date==오늘` 요구 → 휴장일엔 일봉 최신일=전 거래일이라 **재매집·흔들기 suspects 원천 차단 → 자동매매 매수도 살 것이 없음**.
- 미차단 잔여: ① 모니터 매도 경로(stale 가격 손절 판정 → 거래소 거부로 무체결이나 재시도 루프·로그 오염) ② youtong/forecast 표시 stale ③ 연구 cohort 오염. → "높음"으로 수정하되 고치는 방향은 타당.

## 2. 확정 수정 방법 (수정1 채택 + 보정)

수정1의 설계 원칙(접수≠체결 / 주문 전 영구 잠금 / 불명확 시 재주문 금지·수동 확인 / 연구·운영 격리 / 휴장 fail-closed)은 **전부 채택**한다. 이슈별 확정 방법과 보정:

### 이슈 3 — 설치기 안전 기본 (즉시 적용 가능·주문 로직 무접촉)
수정1 §3 그대로: `install_cron_kiwoom.sh`를 권위 SSOT로, `install_cron.sh`는 얇은 wrapper화(인자 없음=DRY, `--live` 명시 시에만 `AUTOTRADE_LIVE=1`), 설치 전후 "실발주 ON/OFF" 경고 출력, 문서 6종 통일. **주의: 현재 운영 crontab은 LIVE가 의도된 상태이므로, 수정 배포 시 회장님이 `--live`로 1회 재설치해야 운영이 유지된다 — 이 전환 절차를 커밋 메시지와 문서에 명시.**

### 이슈 2 — 자동매매 전역 잠금
수정1 §2 채택: `autotrade_common.acquire_execution_lock()`(fcntl `LOCK_EX|LOCK_NB`, 미획득 시 즉시 skip 종료). 실행기·모니터·reconcile 공용. 보정 2건: ① 잠금 파일은 `/tmp/kiwoomnews-autotrade.lock`(publish의 `/tmp/stocknews_publish.lock`과 별개 유지 — 상호 대기 금지) ② 잠금 보유 시간에 키움 API 콜이 포함되므로 모니터 틱이 skip되는 것은 정상 동작임을 로그 문구로 명확화.

### 이슈 1 — 매수 접수·체결 분리 (가장 큰 수술)
수정1 §1 채택: `pending_entries[]`(PREPARED→ACCEPTED→PARTIAL/FILLED/CANCELLED, SUBMIT_UNKNOWN은 수동 확인 전 잠금 유지), 확인 체결수량·체결가중평균만 포지션 전환, `entry_price_verified=false`면 자동 청산 보류. 보정 3건:
1. **§1-1(매수 조회 필드 실측)이 절대 선행** — kt00007 매수측 필드는 미검증. 월요일 장중 소액 1주로 주문번호·체결수량·잔량 필드를 확정하고 fixture 저장 후에만 구현 진행(불확실하면 중단 — 수정1 원칙 그대로).
2. 7/11 매도측 `pending_exit` 패턴과 **동일 구조·동일 함수 재사용**(reconcile 로직 공용화) — 매수·매도가 다른 상태기계를 갖지 않게.
3. `deployed_today`·`already_bought`가 pending_entries도 인식해야 함(미확정 주문 중 동일 종목 재매수 차단·예산 이중 집행 차단) — 수정1 §1-6에 명시돼 있으나 테스트 목록에 예산 이중집행 케이스 추가.

### 이슈 5 — 거래일 검증
수정1 §5 채택: `minute_bars_today_with_meta()`(trade_date 동봉), 중앙 `market_state` 판정(요일만으로 열림 확정 금지·모순 시 fail-closed), radar `STALE_TRADE_DATE` 오류화·publish 빈 게시 금지, 자동매매 독립 재검증. 보정: 브로커 영업일 API가 없거나 불명확하면 **"당일 일봉 존재 + 응답 거래일 일치"를 1차 판정**으로 쓰고 휴장 캘린더는 보조 — 새 외부 의존을 만들지 않는다.

### 이슈 4 — 비게시 연구 cohort
수정1 §4 채택: `research/krx_close_cohort.json`(KRX 마감 전 마지막 성공 스캔 SSOT) + `evaluation/observed_union_next_day.json`(+3/7/11/13/15·−3/5/7/10 터치) + `research_turnover_performance.json`. 보정 2건: ① 회전율 밴드 경계는 7/11 합의안(90/150/200/300 + 110 관찰셀, 사전 등록 문턱 원칙)을 적용 — 수정1의 8분할(0~5…300+)은 관찰 커버리지용 보조 뷰로만 ② API 부하: cohort 익일 평가는 일봉 1콜/종목·일 200콜 수준으로 17:20 잡에 수용 가능하나 최대 호출 수 로그(§4-4) 필수.

## 3. 배포 순서·시점 (수정1의 3→2→1→5→4 채택 + 시점 보정)

| 순서 | 이슈 | 시점 | 이유 |

## 4. 최종 문장

수정1의 진단 5건은 전부 사실이고 계획은 우리 합의 원칙과 정합해 채택한다. 보정은 세 가지다: **① 이슈 5는 기존 `_allow_signal_date` 가드 덕에 크리티컬이 아니라 '높음'** ② 이슈 1은 매도측에 이미 검증된 pending 패턴을 재사용하되 **매수 필드 실측(월요일 장중)이 절대 선행** ③ 설치기 수정 후 운영 LIVE 전환 절차를 명시해 "고치고 나서 자동매매가 조용히 DRY가 되는" 2차 함정을 방지한다.

> **주말에 함정(설치기)과 잠금부터 제거하고, 월요일은 rank4-v1 발효와 매수 필드 실측에 집중하며, 매수 체결 분리는 화요일 이후 검증된 필드 위에서 수술한다.**
